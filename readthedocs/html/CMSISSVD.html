CMSIS-SVD



The CMSIS System View Description format(CMSIS-SVD) formalizes the description of the system contained in ARM Cortex-M processor-based microcontrollers, in particular, the memory mapped registers of peripherals.The detail contained in system view descriptions is comparable to the data in device reference manuals. The information ranges from high level functional descriptions of a peripheral all the way down to the definition and purpose of an individual bit field in a memory mapped register.


The CMSIS-SVD format is based on XML and was influenced by IP-XACT. Due to the much wider scope and complexity of IP-XACT, it was decided to specify a separate format focused and tailored towards the description of the programmer's view of a device.


One CMSIS-SVD file contains the description of a single device. A device consists of a processor and at least one peripheral. Each peripheral contains at least one register. A register may consist of one or more fields. The range of values for a field may be further described with enumerated values.

A device in CMSIS-SVD file provides the outermost frame of the description. The chip described in IDS is converted to device, which includes additional properties like vendor, vendorID and version.

  • address units                     <addressUnitBits>
  • external                             null

Peripheral


A peripheral is a collection of registers. All peripherals of a device are enclosed within the tag <peripherals>. At least one peripheral has to be defined. Each peripheral is enclosed in the tag <peripheral>. A block in IDS is converted to peripheral in CMSIS-SVD.

Cluster


Sections/reggroup/regfiles defined in IDS supported register spec input formats are mapped to clusters in CMSIS-SVD output. These clusters add an optional sub-level within the CMSIS-SVD registers level. They basically describe a sequence of neighboring registers within a peripheral. The address offsets of clusters are relative to the baseAddress of the peripheral. A cluster may contain Multiple <register/> inside it and can also be declared inside a cluster.  Nested cluster express hierarchical structures of registers.

 

Explanation of various nodes in Cluster:


Node 

Description

baseAddress

Address of the section.

dim

The value defines the number of elements in an array of clusters.

dimIncrement

If dim is specified, this element becomes mandatory. The element specifies the address increment in between two neighboring clusters of the cluster array in the address map.

dimIndex

Specifies the substrings that replace the [%s] placeholder within the cluster name. By default, the index is a decimal value starting with 0 for the first cluster element.

name

String that identifies the cluster. Section names are required to be unique within the scope of a peripheral. Specify [%s] for generating an array in the device header file.

description

String describing the details of the section.

addressOffset

Value defining the cluster address relative to the baseAddress defined by the peripheral.

Register


A register is a named, programmable resource that belongs to a peripheral. All registers of a peripheral are enclosed between the <registers> opening and closing tags. Fields


Registers may be partitioned into groups of bits of distinct functionality referred to as field. The field names within a single register must be unique. Only architecturally defined fields shall be described. Any bits not being explicitly described are treated as reserved. They are not displayed in the System Viewer. The following view is a CMSIS-SVD description for fields of register in IDS.

Enumerated Values


Enumerations are used to enhance the readability of source code. Similarly, it can be used by debuggers to provide more instructive information to the programmer, avoiding a lookup in the device documentation. The in-line enums of field in IDS if present take the form of enumerated values in CMSIS-SVD.


      IDS                                  CMSIS-SVD

  • Enums                              <enumeratedValues>
  • mnemonic name                  <name>
  • value                                <value>                              


 <enumerartedValues>
       <enumeratedValue>
                  <name>enabled</name>
                   <value>1</value>
        </enumeratedValue>
        <enumeratedValue>
                    <name>disabled</name>
                    <value>0</value>
         </enumeratedValue>
</enumerartedValues>


Note: 

  • Only enums at field level are supported in this format.
  • Reg_groups in IDS are not supported in CMSIS-SVD
  • Memories in IDS does not contribute to any information for CMSIS-SVD memory-map.
