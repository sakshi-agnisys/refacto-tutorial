Ref support in SystemRDL


This document describes the proposed solution for supporting instantiation of IPXACT blocks/sections/regs in System RDL file. It can be achieved by enhancing “ref” through a UDP named refpath in which user can specify an IPXACT xml file along with component name to be instantiated as shown below:

refpath=”IP-XACTFile.xml:componentName”

where,

IP-XACTFile.xml is the file path to be referred. This path can be either the relative path w.r.t the top file or the absolute path.

ComponentName is the name of component which is to be referred and this is required field.

    • Purpose:
      • The UDP refpath=”IP-XACTFile.xml:componentName” is used to support external files using ref in RDL 2.0. To enable users to instantiate any component from another external file using the ref inside rdl file. 
      • In case of Spreadsheet, UDP refpath value should be:- 

refpath=”Spreadsheet:sheetName:componentname”

    • Scope:
      • IDS will allow users to instantiate any component from another external file only inside the addrmap of the current rdl file.
      • There is no limitation for type of files, Users can ref any type of files that IDS support like:- .rdl, .xml(IP-XACT), .xls, docx, .yml, etc.
      • Users can apply any property or pass parameter using property that can be applied on the ref component.
    • Validations:
      • Any of the components can contain either another component or refpath UDP. For example, if an addrmap can contains either reg/ regfile components/elements or refpath property.
      • The relative path specified in ref path is with respect to the path of RDL file in which this property  is being specified.
      • Order of the refpath property value is important. First value before the ‘:’ will be considered as ref file path and the value after ‘:’ will be considered as component name which is to be referred.
      • In case of Spreadsheet input as ref, first value before the first  ‘:’ will be considered as ref file path and the value after first ‘:’ will be considered as sheet name which has the referred component and the next value after second ‘:’ will be considered as component name which is to be referred.
      • Users have to define refpath UDP at top of the input file or inside properties file.
      • Users can apply any property or UDP on the component which has a ref component.
      • Inside an addrmap/regfile/reg component there must be only one ref component. If users want to instantiate multiple ref components from different files, then they can create anonymous components.

Following are the scenarios, along with the example codes, in which this UDP is going to be supported:


  1. Block inside a chip with only ref content.

property refpath { type = string; component = addrmap;};


addmap chip {

addrmap block1 {

refpath ="block1.xml:block1";

};

block1 b1;

addrmap  {

refpath = “block2.xml:block2”;

} b2;

};


  1. Blocks containing regfile or register.


property refpath {type = string; component = addrmap|regfile;};

addrmap block1 {

addrmap regfile1 {

reg r1 {

field {}  f1;

};


r1 r1;

r1 r2;

                       };


regfile {

refpath=”block1.xml:section1”;

} regfile2;


reg reg {

field { } f1;};

} reg1;

};


  1. Blocks containing repeated registers inside repeated sections and ref.


property refpath {type = string; component = addrmap|regfile;};

addrmap agnisys_tc_csr {

desc = "GMC Level RDL Description";

regfile reggroup1 {

reg GC_CFG {

desc = "";

regwidth = 32;

field {

hw = r;

sw = rw;

desc = "";

} foo;

};

                GC_CFG GC_CFG[5];

};

reggroup1 section1[5];


regfile reggroup2 {

refpath=”regfile.xml:reggroup2”;

};

reggroup2 section2[5];

};
