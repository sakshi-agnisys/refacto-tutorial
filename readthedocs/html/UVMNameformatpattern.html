UVM Name format pattern

(Compatible with IDS version 5.14.24 and above)


These properties have the capability to customize the name of the class and handle name of the class  for uvm output.


Property name

Applied on 

Hierarchical

Description

uvm.name_format

Chip/block/section/memory/reg

YES

Changes the class name

uvm.handle_name_format

block/section/memory/reg/field

YES

Changes the handles name

uvm.reg_name_format

Chip/block/section/memory/reg

YES

Changes the register name

uvm.enum.name_format

Chip/block/section/memory/reg

YES

Changes the enum label's name



uvm.name_format :   <prefix string> %c%b%s%m%r <postfix string> ,  camelcase


uvm.handle_name_format :   <prefix string> %c%b%s%m%r%f%p <postfix string> ,  camelcase


uvm.reg_name_format :   <prefix string> %c%b%s%m%r <postfix string> ,  camelcase


uvm.enum.name_format :   %c%b%s%m%r,  camelcase


Prefix string: This prefix string will add as a prefix to the class name generated. This string can be multiple words with/without space separated. 

Postfix string: This postfix string will append in the suffix to the class name generated. This string can be multiple words with/without space separated.

camelcase: To generate the class/handle name in camel format.  


Note: 

While writing the name format pattern, it must be emphasized that there is a white space after the <prefix string> and before the <postfix string>.


  • %c : Here 'c' is referring to CHIP name. It will add the chip name in the class name generated.
  • %b : Here 'b' is referring to BLOCK name. It will add the block name in the class name generated.
  • %s : Here 's' is referring to SECTION/REGGROUP name. It will add the section/reggroup name in the class name generated. If there are many levels of section, then the user can choose which level of section name to be include in the name generation. The syntax is “%s1/%s2” here '1' , '2' is the level of the section. The syntax “%s” always choose the current level name for section class name and the parent section name for down level classes. 
  • %m : Here 'm' is referring to MEMORY name. It will add the memory name in the class name generated.
  • %r : Here 'r' is referring to reg name. It will add the register name in the class name generated.
  • %p : Here 'p' is referring to present node(which can be Chip/block/section/memory/reg/field) depending on the node from which the property has been applied.

This property changes the class name in the generated UVM Output.

2. uvm.enum.name_format


For specificifying the name of enum labels, we have a property in C Header namely, {cheader.enum.name_format} for the enum name format. Now, the same property has been supported in UVM RAL i.e., {uvm.enum.name_format}.

For Example:

{uvm.enum.name_format=%c%b%s%r%f%e, camelcase}

where,


camelcase: To generate the enum name in camel format.

  • %c : Here 'c' is referring to CHIP name. It will add the chip name in the enum name generated.
  • %b : Here 'b' is referring to BLOCK name. It will add the block name in the enum name generated.
  • %s : Here 's' is referring to SECTION/REGGROUP name. It will add the section/reggroup name in the enum name generated.
  • %r : Here 'r' is referring to reg name. It will add the register name in the enum name generated.
  • %f : Here 'f' is referring to field name. It will add the field name in the enum name generated.
  • %e : Here 'e' is referring to enum name. It will add the enum name in the enum name generated.


Default enum label name generation will get override by the property and will generate according to it.

On Applying Property (i.e. uvm.enum.name_format=%c%b%s%r%f%e, camelcase):


Example: IDS-NG        IDS-Word        IDS-Excel        SystemRDL


IDS-NG Register View



IDS-NG Spreadsheet View

4. “inst_name” Property


“inst_name” property is used to append the block name in UVM to the name given with inst_name. When a large number of parameters are used in a case, block names are used to get appended with all the parameters. Which can bring difficulty in reading the block name. Thus inst_name was introduced to change the block name with the name you'll give with this property.


SystemRDL Input:


property inst_name {type = string; component = addrmap|regfile|reg;};


reg reg1 #(longint unsigned WIDTH = 31,

                        boolean ABC = true){

        field {} f1[WIDTH:0]=2;

};

        chip = true;

5. uvm.handle_name_format


SystemRDL Input:


property uvm_handle_name_format { type = string; component = addrmap | regfile | reg | field | mem; };


 addrmap block1 {

    uvm_handle_name_format = "abc %r";

    reg reg1 {

       regwidth = 32;

      field {

        hw = rw;

        sw = rw;

      } f1[31:0] = 32'h0;

    };

    reg1 reg1 @0x0;

 };


UVM output:


class block1_block extends uvm_reg_block;

    `uvm_object_utils(block1_block)


    rand block1_reg1 abcreg1;


    // Function : new

    function new(string name = "block1_block");

        super.new(name, UVM_NO_COVERAGE);

    endfunction


    // Function : build

    virtual function void build();

        //define default map and add reg/regfiles

        default_map= create_map("default_map", 'h0, 4, UVM_BIG_ENDIAN, 1);


        //REG1

        abcreg1 = block1_reg1::type_id::create("abcreg1");

        abcreg1.configure(this, null, "abcreg1");

        abcreg1.build();

        default_map.add_reg( abcreg1, 'h0, "RW");


        lock_model();

    endfunction


endclass
