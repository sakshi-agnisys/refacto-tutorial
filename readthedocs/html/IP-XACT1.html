IP-XACT

IP-XACT GENERATION: 


IP-XACT describes an EXtensible Markup Language (XML) data format and structure, documented with schema for capturing the meta-data, which documents design intellectual property (IP) used in the development, implementation, and verification of electronic systems.

 


This standard is intended for use by ELECTRONIC DESIGN AUTOMATION(EDA) and ELECTRONIC SYSTEM LEVEL(ESL) tools. It was created and is owned by the Accellera.


The primary goals of the standard are:

    • To enable IP vendors to provide a single description of their components to all of their customers, regardless of the design language or tools

that they use

    • To enable developers to transfer designs between environments that use different design languages.


IP-XACT includes an XML schema that defines a number of document types, and a set of semantic rules that describe the relationships between those documents. The most important types of documents are:

    • Design documents.
    • Component documents.
    • Bus definition documents.
    • Abstraction Definition documents.



IDesignSpec™ can both read and write IP-XACT files in all its flavors : IDSBatch, IDS-Word, IDS-Excel.

An IP-XACT component is the central placeholder for the object‘s meta-data. Components are used to describe cores, peripherals, and buses.

The top-level described in register spec using IDS is converted to or ipxact:component or spirit:component, which includes additional properties like vendor, library name and version.


 


<ipxact:memoryMaps> or <spirit:memoryMaps> is an unbounded list of memoryMap elements, which is contained inside <ipxact:component> or <spirit:component>. The chip in IDS is converted to element <ipxact:memoryMap>  or <spirit:memoryMap> inside <ipxact:component> or <spirit:component>


The addressBlock element describes a single, contiguous block of memory that is part of a memory map. A Block inside IDS is converted to <ipxact:addressBlock> or <spirit:addressBlock> in IP-XACT. The offset is mapped to < ipxact:baseAddress> or <spirit:baseAddress>and the size is mapped to < ipxact:range> or <spirit:range>.

Like chip can contain any number of blocks, <ipxact:memoryMap> or <spirit:memoryMap> (which is contained inside <ipxact:component> or <spirit:component>) can contain any number of <ipxact:addressBlock>  or <spirit:addressBlock>’s .

ipxact_exclude_vendorextensions

IDesignSpecTM supports the “ipxact_exclude_vendorextensions=true” property that is applied only on the top node. This property is used for excluding "spirit:vendorExtensions" nodes from the IP-XACT output.



Example


IDSWord



IDSExcel



SystemRDL


property ipxact_exclude_vendorextensions{type = boolean; component = addrmap;};

addrmap chip {

        ipxact_exclude_vendorextensions = true;

        addrmap block1 {

                reg reg1 {

typeIdentifier for IP-XACT output 

Use of typeidentifiers in IP-XACT is to provide a standardized and structured way to categorize and classify IP components within an IP-XACT description.Type identifiers are typically used to categorize IP components or elements based on their functionality, purpose, or characteristics. They help classify and organize the components in a hierarchical manner, allowing for easier navigation and understanding of the IP-XACT design hierarchy. Let's take a look at an example to understand this. Consider the following input:                        


SystemRDL:

             

property chip { type =boolean ; component=addrmap ; } ;

reg reg1{

    field{  

        sw=rw;

        hw=rw;

     }f1[31:0];

    };

reg reg2{

    field{  

        sw=rw;

        hw=rw;

     }f1[31:0];

    };


regfile sec1{

  reg1 reg11; 

  reg2 reg22;

};


addrmap block1{

     reg1 reg1_1;

     reg1 reg2_2;

};

addrmap block2{

     sec1 sec1_1;

     sec1 sec2_2;

};


addrmap top{

    chip=true;

    block1 block1_1;

    block2 block2_2;

};


Command line:  idsbatch input.rdl -o ipxact1_5 -dir ids
