I2C-Beta

I2C (Inter Integrated Circuit) - Beta Version


IDS supports the I2C bus protocol. Various operating modes are implemented.  It contains a comprehensive introduction to the I2C-bus data transfer, handshaking and bus arbitration schemes.This provides an information about, How a user can easily communicate with IDS serially using I2C initiator at the software side. Off-course there are some initial configurations before starting with I2C: IDS provides a way to configure those initials at the beginning itself, It provides a user to configurable bus select option, if I2C then selecting device addressing mode and target address. 

The I2C provides an interface between the initiator and other devices compliant with Philips Semiconductors Inter-IC bus (I2C-bus) specification version 2.1 and connected by way of an I2C-bus. Most TI devices (c5000, c6000, OMAP, Davinci, etc.) use same underlying modules.


Supported Bus Features


  • Compatible with Philips I2C Standard. Only two bus lines are required; a serial data line (SDA) and a serial clock line (SCL).
  • Single initiator operation. 
  • Software programmable clock frequency, Generally 3 Transmission speeds: Normal, Fast, and High Speed.
  • Software programmable acknowledge bit.
  • Interrupt or bit polling driven for each byte.
  • Start, Stop, Repeated Start detection.
  • Supports 7 bit addressing mode, with configurable address masking in target mode.
  • Supports strict I2C reserved rule, Detects general call addresses.
  • Support for IDS compatible byte format transfer for Register address.
  • Bus busy detection.


Features Not Supporting


  • High speed and Ultra fast- mode
  • CBUS Compatibility mode.
  • Bus Arbitration
  • Arbitration Lost interrupt
  • Multi initiator operation
  • 10 bit addressing mode.

SDA and SCL signals


The I2C interface uses bidirectional serial data line (SDA) and a serial clock line (SCL) for data transfers. All devices connected to these two signals must have open drain or open collector outputs. Both lines must be pulled-up to VCC by external resistors.

Data on the I2C-bus can be transferred at rates of up to 100 kbit/s in the Standard-mode, up to 400 kbit/s in the Fast-mode, up to 1 Mbit/s in Fast-mode Plus, or up to 3.4 Mbit/s in the High-speed mode. The bus capacitance limits the number of interfaces connected to the bus.

For a single initiator application, the initiator’s SCL output can be a push-pull driver design
if there are no devices on the bus which would stretch the clock.

To solve that user needs to add tri-state buffers for the SCL and SDA lines at the top hierarchical level.

Verilog code:

assign scl = scl_padoen_oe ? 1’bz : scl_pad_o; assign sda = sda_padoen_oe ? 1’bz: 

sda_pad_o; assign scl_pad_i = scl; 

assign sda_pad_i = sda; 

(These are padding signals are the external port connections for inserting buffers.)


Prescaling of SCL line

Due to the structure of I2C interface, the core uses a 5*SCL clock internally. So the speed of the I2C bus must be programmed to this 5*SCL frequency (minus 1). 

Prescale value = (input SCL MHz / 5*Desired SCL KHz) – 1

Generate START/STOP condition

The start condition is generated by initiator. The logic for Start condition is generated by pulling SDA line low when SCL is high. This shows an I2C bus has been entered into busy mode and initiator now can send serial transactions to the target.

The stop condition is also generated by initiator, when it assumes that the serial data transaction has been completed by pulling SDA line high when the SCL is high. This indicates that an I2C bus now is in Free State any other initiator can communicate with the bus.

Data Validity

The data on the SDA line must be stable during the HIGH period of the clock. The HIGH or LOW state of the data line can only change when the clock signal on the SCL line is LOW.

DATA Validity


7-bit target Address and RD/WR bit Format

After the START condition (S), a target address is sent. This address is seven bits long followed by an eighth bit which is a data direction bit (R/W) — a ‘zero’ indicates a transmission (WRITE), a ‘one’ indicates a request for data (READ).

A data transfer is always terminated by a STOP condition (P) generated by the initiator. However, if a initiator still wishes to communicate on the bus, it can generate a repeated START condition (Sr) and address another target without first generating a STOP condition. Various combinations of read/write formats are then possible within such a transfer.

Configurable Addressing

IDS provide a facility to configure 7-bit addressing of a particular device before start generation of I2C Interface with IDS. This addressing follow all the rules and regulations complaint with Philips Semiconductors Inter-IC bus (I2C-bus) specification version 2.1. 

IDS can support target addressing formats: 7-bit  

User needs to select a mode of target select first by just click an option available in IDS configure window or he/she may add an property to specify an mode of target address followed by a particular target address value.


Arbitration Procedure:

CLOCK Stretching:

Target devices can use the clock synchronization mechanism to slow down the transfer bit rate. After the initiator has driven SCL low, the target can drive SCL low for the required period and then release it. If the target’s SCL low period is greater than the initiator’s SCL low period, the resulting SCL bus signal low period is stretched, thus inserting wait states.

IDS widget can handle this condition if there is a wait state from an internal signal rd_wait during read operation, initiator will drive SCL low for required period of time.


 8-bit of Register Address:

After the target address being acknowledged, User needs to send a byte that is a specified IDS register address which is byte addressable by the increment of four bytes for each register.

If there is a memory specified, this controller will take care of an appropriate address calculation or an offset value compatible with IDS requirements.

Because the default address of an IDS registers is 32 bit, widget will split this address into separate bytes which are followed by one after one covering complete register by using repeated Start logic. 

   

Sending/Receiving Data Bytes:

Once a complete address has been acknowledge by the target then initiator can send the data into corresponding Register.

If the initiator initiates a read transaction and there is a case of latency from the target means IDS rd_wait signal is enable, then target can force initiator into wait state by holding SCL line Low until the target is ready for the next byte transfer in a type of
handshake procedure. This is also called Clock Stretching.
