5. Project Based configuration 


In IDS-NG, currently users can set configuration for the file not for the entire project, To set configuration for the entire project that can be inherited and overridden by the individual file from the same project. This will also enable some features like setting different configuration for individual top files when it will be generated from the file, and can also set different configuration from project configuration when users want to generate from the project. In addition, users will be able to treat the project as a whole, not just separate individual files. Users will be able to generate output for the whole project. configuration will be set once at the top level or individually at a file level. It will be possible to override the properties from the top. 

A project may have several “top-level” files. Command line options or top properties added to the project will get added to all top level files. Users will be able to set top level files in a project. 


Objective:


Objective of this strategy to give user flexibility that users can generate all from the project. This will enable enterprise feature like following:


    • Users can set different configuration for different IDSNG files and can generate from projects and individuals as well.
    • Users can inherit some configuration settings from project configuration to file configuration and can generate outputs.
    • Users can override some project configuration settings for the individual IDS NG files that can generate outputs.
    • If the user generates outputs using project configuration then there will be an option to select and choose one or multiple top files.
    • If the users choose multiple top files and generate output using project configuration then it will generate output simultaneously for all selected top files at the same time.


How to use:


    • Create many specifications in a project and check all the backward compatibility.
    • Select the project and open project configuration and then mark some specification as top specification and generate on right click on project and check if the file is generated correctly or not.
    • Open configuration and mark more than 2 spec as top and select some top spec to use file configuration and then try to generate a project and then check whether output is generated according to their configuration setting or not.

9. IDS-NG Update Param Enhancement

Introduction

Users can switch between reg view and param view. Param View is based on a yaml file (.yml).


IDS-NG, Param view has the information about the registers and their fields. It does not have the information about the block, chip or any other template used in IDesignSpec™.


Also, while hovering over fields in the param view, IDS-NG gives information like, name, size, software/hardware accesses and default values of the fields.


Users can move the fields of registers from one address location to another by dragging them and corresponding changes reflected in the reg view. Changes made in the reg view or spreadsheet view are also reflected back in the param view. Now, the param view is editable. We can add, delete or paste registers in the param view. Also, if we do any modification in the param view, then it will get reflected in the register view and the other views.

A. Update Param

IDS-NG is introducing an Update Param option. This option is different from the Import Param option. The Update Param option allows to smoothly update param data from another yml or yaml file to an existing param view without losing any data. The Update param works well even if all or a specific cell is locked it simply adds a new param to new rows.


                               

           

                                       


B. Added Param


IDS-NG supports the Added Param feature along with the Update Param option. 

This feature displays the available param name list which is added through the Update Param option and these param can be easily added  by clicking on the checkbox. Unchecked box denotes available but unadded param data and checked box denotes param that have been added in param view.

12. Incremental Output Specifications


Incremental generation is implemented to reduce generation process. Basically, this functionality focuses on generating only those outputs which are required. 


Implementation: 

Following are the steps to implement. 

  • Diff current and previous annotated file.
  • Define a SET_OF_RULES that indicate which IP should generate.
  • Add “generate=false” attribute to annotated XRSL that should not generate. If “generate” attribute not present or “generate=false”, this should consider as not generate.  
  • Put “generate=false” checks on output generating functions.


Implementation Flow:


SET_OF_RULES:

  • Check only when top is Board or Chip.
  • Add ‘generate=true’ in AXRSL that indicate that specific element should generate.
  • Implement the Phase 1 rule.


After analysis and discussion (reference below matrix), we can implement this into two phases as below:


Phase 1:


If there is any diff in Element (block/chip/board) generate all outputs for that Element. 


Phase 2:

Consider the type of changes for specific Element and then generate accordingly.
