Hierarchical Decode


There are two ways to decode addresses for the IP blocks. One is the absolute decode where all address bits are decoded at each block. This is currently by default in IDesignSpec™. The other way is to do hierarchical decode in which the address bits are divided into two parts, the MSB part that is used to decode the block and the LSB part that is used to decode the register being addressed once the block address (the MSB part) is selected.


Hierarchical decode is advantageous because of the following reasons:

  • The entire address bus need not be routed to all the blocks in a chip, this reduces the routing congestion.
  • Since decode is happening only at one place and select lines are used to signal the selected blocks, the power consumption is low.
  • A select line is used for each block that indicates that the block’s registers are selected, this has the advantage that addresses offset does not need to be passed as a parameter to the block. This, in turn, has the advantage that the same physical block can be reused multiple times.


IDesignSpec™ supports this with property “hierarchial_decode=true”.  The property is applied at the chip as illustrated below:

Hierarchical decode enhancement to support multicast/broadcast feature


The property ‘hierarchical_decode’ has now been enhanced to support mapping a target select to certain address bits in the bus interface (say, APB) providing a special decoder that allows multicasting/broadcasting while accessing the targets for a write transaction. 

The enhanced property can be defined in the  following way: - 

 {hierarchical_decode=<base_address>,<target_slector_bits>,<register_address_bits>,<no_of_default_blocks>}

  • The base_address represents the base address of the multicast/broadcast region in the address map.. 
  • The target_slector_bits represents the number of targets in the multicast/broadcast region with multicast/broadcast write access.
  • The register_address_bits  represent the component(register) selector bits which are the don't care bits.
  • The no_of_default_blocks represents the number of default blocks in the multicast/broadcast region starting from the base address.

 

 

Note: In the proposed flow, only one multicast/broadcast region in the defined registermap. And, this flow is currently supported with the APB bus interface.


Block diagram:



Figure (i) : Representation of Selector Logic for input {base_address(0x4123_0000), no of targets(3), component(register) selector bits(13),no of default block(1)}

In the figure(i), the target select addressing might also be done in other areas, e.g. using bits 13 to 15, leading to a target address space of 13 bits (as bit 0 to 12 are not included in the special decoding mechanism and, thus, are decoded by the target logic).

Note: 

  • The multicast-mode shall be allowed only for WRITE transactions.
  • A READ transaction from a single target shall be allowed, but a READ transaction from a multicast address will report an error to the bus.
  • Invalid addresses inside the multicast/broadcast region (w.r.t blocked base address area) within the chip are ignored.
