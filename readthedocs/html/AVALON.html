AVALON



IDS supports Avalon-MM bus both in Verilog and VHDL. All the transactions to the registers pass through this Avalon widget. Bursting is supported. Also, in case of access to empty addresses, a proper error response is generated.

Data for write transfer. The width must be the same as the readdata if both are present.

Asserted to indicate a read transfer. If present, read data is required.

(default is 32)

target

The readdata is driven from target to the initiator in response to a readdata.

byteenable

configurable

(default is 4)

initiator

Each bit in byteenable corresponds to a byte in writedata. During writes, byteenable specify which bytes are being written to; other bytes should be ignored by the target. Target that simply return readdata with no side effects are free to ignore byteenables during reads.


When more than one bit of byteenable signal is asserted, all asserted lines are adjacent. The number of adjacent lines must be a power of 2, and the specified bytes must be aligned on an address boundary for the size of the data.


 For example, the following values are legal for a 32 bit target:


  1111 writes full 4 bytes

  0011 writes lower 2 bytes

  1100 writes upper 2 bytes

  0001 writes byte 0 only

  0010 writes byte 1 only

  0100 writes byte 2 only

  1000 writes byte 3 only

waitrequest

1

target

Forces the initiator to wait until the system interconnect fabric is ready to proceed with the transfer. 

readdatavalid

1

target

Used for variable latency, pipelined read transfers. Asserted by the target to indicate that the readdata signal contains valid detail response to a previous read request. A target with readdatavalid must assert this signal for one cycle for each access it has received.

burstcount

1-11

initiator

Used by bursting initiator to indicate the number of transfers in each burst. The value of the maximum burstcount parameter must be a power of 2. A burstcount interface of width <n> can encode a max burst of size 2(<n>-1). For example, a 4-bit burstcount signal can support a maximum burst count of 8. The minimum burstcount is 1.

beginbursttransfer

1

initiator

Asserted for the first cycle of a burst to indicate when a burst transfer is starting. This signal is deasserted after one cycle regardless of the value of waitrequest.

response

2

target

The response signal is  signal that carries the response status. 

Note: Because the signal is shared, an interface cannot issue or accept a write response and a read response in the same 

clock cycle.

 • 00: OKAY—Successful response for a transaction.

 • 01: RESERVED—Encoding is reserved. 

 • 10: TARGETERROR—Error from an endpoint target. Indicates             an unsuccessful transaction. 

 • 11: DECODEERROR—Indicates attempted access to an                 undefined location.


For read responses:

• One response is sent with each readdata. A read burst length of N results in N responses. It is not valid to produce fewer responses, even in the event of an error. It is valid for the response signal value to be different for each readdata in the burst. 

• The interface must have read control signals. Pipeline support is possible with the readdatavalid signal.

• On read errors, the corresponding readdata is "don't care". 


For write responses: 

• One write response must be sent for each write command. A write burst results in only one response, which must be sent after the final write transfer in the burst is accepted.

writeresponsevalid

1

target

When asserted, the value on the response signal is a valid write response. 

Writeresponsevalid is only asserted one clock cycle or more after the write command is accepted. There is at least a one clock cycle latency from command acceptance to assertion of writeresponsevalid.
