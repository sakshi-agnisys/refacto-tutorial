Reset signal

Introduction:

In digital system, a reset is action to clear any pending process and brings a system to normal condition or an initial state, usually in a controlled manner.


In IDS, we use reset signal to reset design circuit with two type:


1. Soft reset: The soft reset is primary reset, which is used to reset some general setting or to fix some low-level issue (problem, bug). It fixes minor issue. In IDS soft reset signal is defined by the property ‘resetsignal’.


2. Hard reset:  The hard reset affects the OS level setting and data. A system is given a hard reset when the reset button in pressed.  In IDS hard reset is implemented as an input signal called ‘reset_l’.


Here are the properties that are related to Resets in IDS.

  • reset_type: Define type of ‘reset_l’. [sync, async] {C, B, R, F}
  • reset_level: Define level of ‘reset_l’. [low, high] {C, B, R, F}
  • hard_reset: Enable/disable ‘reset_l’. [true,1/false,0] {R, F}
  • has_reset: Enable/disable reset from UVM. [true,1/false,0] {F}
  • default_reset_name: Change name of ‘reset_l’.[“string”]{C, B}
  • rtl.default: Change default value. [signal_name] {R, F}
  • resetsignal: Soft reset. [signals and/or values] {R, F}
  • field_reset: Apply reset on(default). [true/false] {S}
  • level: Level for soft reset. [low, high] {S}
  • type: Type for soft reset. [sync, async] {S}


Here []-Possible value, {}- Possible place, C-chip, B-block, R-register, F-field, S-signal


We can also use soft resets by combining all the reset signal properties with corresponding default value, as given below:


{resetsignal=rst_sig1:d_val:type:lvl}

                   

where - ‘rst_sig1’ is the soft reset signal, described in the signal table ‘d_val’ is default value of the corresponding soft reset signal ‘type’ is type(sync, async ) of the corresponding soft signal ‘lvl’ is level(low, high) of the corresponding soft signal.


  Note:

  • If we do not want to provide any property value of the soft signal, leave it empty between the ‘:’, IDS will take the default values.

              e.g., {resetsignal=rst_sig1:d_val::lvl} 

Here in the above soft reset signal 'rst_sig1', the type of the reset signal will be sync (by default) as its value is left empty while declaring the signal.

  • Backward compatibility supported.
  • Exists in IDS version 6.4.24.0 and later, in RTL.


reset_level


This property specifies the level of the reset for the registers.

Reset Signal In IP-XACT Output


To support Reset Types(Soft Reset and Hard Reset) in IP-XACT output. If a register has multiple fields and those fields are assigned with multiple resets(hard reset and soft reset).

The idea is to create a register reset value and mask for each of the reset in IP-XACT vendorExtension.

Example: Below are the examples/Prototype of input files and the output IP-XACT files with reset types.

1. In the below example block ”blockA” is containing a register “regA” with fields FA, FB, FC, FD, FE.

  • Field FA, FC, FD, FE containing the property “resetsignal=SoftReset”.
  • Field FB containing the property “hard_reset=false” which will remove the hard reset functionality from the field.
  • So, the register has two reset types one is hard reset(default) and soft reset with signal “SoftReset”
  • Hard reset is removed from the field “FB” then the field “FB” will be masked out from  the register “regA” hard reset “reset_l” mask and serset value. 

2. After Masking out the “FB” the reset mask will be :  0xFE0FFFFF and reset value will be : 0x000182C9

  • Reset signal property applies soft reset functionality to a field. The field “FB” in register “regA” is not applied with this property and will be masked out. The reset value will be  “0x000182C9” and mask 0xFE0FFFFF

Multiple Reset Value for Register Fields(Concatenation operator)


In case of repeating registers, if different “default” values of fields of each instance that register is to be specified, then following mechanisms can be used.


See example below. MODE3 field is set to "2" in all repeating instances. MODE2 filed has each default value explicit specified for each of the 32 instances of the register field. MODE1 and MODE4 uses the concatenation operator which is the most compact and sophisticated method is IDS for specifying defaults. 




NOTE:


Only three type of concatenation is supported in IDesignSpec™

  • Simple-  2{3} = Default is 3 for first instance and 3 for second instance of field.
  • Level one- 5{1},27{0} = Default is 1 for first 5 instances and 0 for next 27 instances of field.
  • Level Two- 2{8{1},8{5}} = Default is 1 for first 0 to 7 and 16 to 23 instances. And 5 for 8 to 15 and 24 to 31 instances of field.

No parameter or enum is supported inside IDS concatenation operator.


As of version 6.4 the concatenation feature is only supported in Verilog and SV outputs.

IDS Concatenation Operator


In case of large repeat on registers and reg group, if user wants different “default” values of fields of each instance of that register and reg group, then user can use following methods of IDesignSpec™ for specifying the default value.  

 

The most compact and sophisticated method is IDS Concatenation operator. The following example shows how to use IDS Concatenation operator and other methods of specifying default values. 





Note: The number of default values should be equal to the number of register and reg group together. One default value for each instance.

Power on Reset (PoR):


The power-on reset is an electronic device that detects the power applied to the system and generates a reset impulse that goes to the entire circuit placing it into a known state (idle). It is RC circuit, charging capacitor to control reset signal voltage.


It is a technique to check, if all registers which have reset signal (soft reset), are reset. PoR brings the system from any state to normal state.


Why:


  • To provide constant voltage at reset signal.
  • To start the system in desired and error-less state.
