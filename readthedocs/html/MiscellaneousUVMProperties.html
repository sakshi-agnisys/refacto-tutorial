Miscellaneous UVM Properties

uvm_reg_access


If all fields in a register are software readable then the software access of register will be "RO", similarly  for writeable fields. In case fields in a register are readable as well as writeable or have other special access, then register access will be "RW".

IDS-NG Spreadsheet View

uvm.alter_access


A new property uvm.alter_access=true has been implement to alter the UVM accesses. This property will be applied on the top level chip/block of the hierarchy. The problem which we currently face is that if, register access is “wo” then there is a restriction on fields to be wo/rw.  So, in case of field access w1t/w1c and register access is “wo” there will be a restriction on field. To solve this problem w1t/w1c access is converted to “wo” access in UVM, and callback is used to toggle/clear value. 


The callback example has been shown below:


Example: IDS-NG        IDS-Word        IDS-Excel        SystemRDL


IDS-NG Register View



IDS-NG Spreadsheet View



SystemRDL

 property sw_access {type = string; component =mem|reg; };  

              

  addrmap block1 {


  uvm_alter_access = true ;


  reg reg1 { 

     sw_access = "wo" ;

     regwidth = 32;

    field {

      hw = rw;

      sw = w;

      onwrite=woclr;

    } f1[31:0] = 32'h0;


  };

uvm_contxt


When we have a register file with registers further having multiple instances and for instance override, we pass the path name to the create function.

In other words, we can say that to override the values to be programmed to the register field through the UVM factory but not using the override by type but override by instance.

In that case we have a property “uvm_contxt” . Using this, we can give the path name to create the function.


SystemRDL Example:



  `ifndef IDS_UDP

           

     property uvm_contxt { type = string; component = regfile | reg; };

     

 `endif

 

addrmap Block1 {

  name  = "Block1 Address Map";

   

  reg Reg1 { 

     uvm_contxt = uvm_test_top:get_full_name();

     regwidth = 32;

     count =3;

  uvm_contxt = "uvm_test_top:get_full_name()";

    field {

      hw = rw;

      sw = rw;

      

    } F1[31:0] = 32'h0;


  };

 

  Reg1 Reg1[4] @0x00;


};

Support of Signals in reset/default in UVM RAL


“Signals” create, named external ports, in the block or chip level ,and connect certain internal design signals to the external world in RTL design.

As per SystemRDL standard, signals could be applied to reset values of a field and  also the values for reset through dynamically.

type = "nomem"


In IDS, by default external regfile which contain a single register inside it, it is treated as a memory and the class extends from uvm_mem. IDS property “type = nomem” which is identical to "map_type=nomem" UDP in SystemRDL. It is applied on the external regfile which contains only a single register and the regfile and register inside register classes extend with uvm_reg_file and uvm_reg respectively. It is only supported in UVM.

uvm_add_regmap_offset


This property is used with the “uvm_add_regmap” property, using this we can change the offset of the regmap generated by “uvm_add_regmap”. The syntax to use this property is uvm_add_regmap_offset=”<name>:<base address>”.

Property "uvm_map_access" and "uvm_map_notpresent"


Property uvm_map_access is applied at the component to change the register sw access of user defined map or default map and uvm_map_notpresent property is applied at the component to remove that register from the particular user defined map or default map and both the properties should be applied with uvm_add_regmap.
