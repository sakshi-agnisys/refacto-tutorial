Using Original Name as Primary

This document describes the changes in naming convention of all SystemRDL components “addrmap, regfile, reg” to generate the output collateral for UVM and C Header as required by the user.

On instantiating the SystemRDL components multiple times in a hierarchy, the requirement comes to how many collateral will be getting generated for those instances. This can be achieved by identifying the identical and unique instances. In SystemRDL, unique instances are created using one of two techniques as given below:

  • Parameter Overriding: In this technique, naming convention of generated collateral is decided based on the overridden value of RDL parameters.
  • Dynamic assignment of properties and UDPs: In this technique, naming convention is decided based on the ordering of components in the IP. Using the ordering of components, the first instance has the original name and the rest of the instances have the instance name.

Conventionally, whenever a component “addrmap, regfile, or reg” is instantiated multiple times in a hierarchy then the instance name of instantiated component is primarily used for generated collateral. While the requirement is to provide a mechanism to selectively use the original name of the component instead of the instance name for the generated collateral.

Parameter Overriding:


Example:


addrmap block_csr1 #(boolean param = false) {

.........

};


addrmap block_map #(boolean param = true) {

block_csr1 #(.param(param)) r1;

block_csr1 #(.param(param)) r2;

};


Case - I


If the value of parameter “param” for both instances “r1” and “r2” is same as given below:


addrmap block_map #(boolean param = true) {

block_csr1 #(.param(true)) r1;

block_csr1 #(.param(true)) r2;

};


Current Output:


Only one collateral is generated for “block_csr1” using the instance name “r1”.


Required Output:


Only one collateral should be generated for “block_csr1” using the original name along with the value of the overridden parameter e.g., “block_csr1_true”. 


Case - II


If the value of parameter “param” for both instances “r1” and “r2” is different as given below:


addrmap block_map #(boolean param = ttrue) {

block_csr1 #(.param(true)) r1;

block_csr1 #(.param(false)) r2;

};


Current Output:


Two collaterals are generated for “block_csr1” using the instance name “r1” and “r2”.


Required Output:


Two collaterals should be generated for “block_csr1” using the original name “block_csr1_true” and “block_csr1_false”.


Note: Why the value of overridden parameters used (for naming as described in Case - I and Case - II) instead of ordering of components?

Case - III and Case - IV answer this question.


Case - III


If addrmap “block_csr1” is instantiated inside multiple components in a deep chip-in-chip hierarchy of components as given below:

block_csr1 #(.param(false)) r2;

Dynamic Assignment of properties and UDPs

Example:

addrmap block_csr1 {

.........

};


addrmap block_map {


block_csr1 r1;

r1->name = "First instance";


block_csr1 r2;

r1->name = "Second instance";


};

The generated collateral will be based on case III the mentioned above.

Note: The same convention is followed for all the System RDL components. This requirement originated using the re-usability of RDL components therefore this change will be done in SystemRDL compiler only which leads to update in all generated IDS collateral.
