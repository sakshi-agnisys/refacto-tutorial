C Header API code


#include <stdbool.h>

#include "C:\Users\User_Name\Documents\.IDS\Cblock.h"


#define  CBLOCK_REGA_VAL (*(volatile hwint32*) Cblock_regA_ADDRESS)

#define  CBLOCK_REGA_READ(data) data=(CBLOCK_REGA_VAL)

#define  CBLOCK_REGA_WRITE(data) (CBLOCK_REGA_VAL)=data


/*---------------------------- Read Function for field : Fld ---------------------------------*/

hwint32 cblock_rega_fld_read()

{

    hwint32 read_data;

    CBLOCK_REGA_READ(read_data);

    return ((read_data & CBLOCK_REGA_FLD_MASK) >> CBLOCK_REGA_FLD_OFFSET);

}

/*---------------------------- End of Read Function for field : Fld --------------------------*/



/*---------------------------- Write Function for field : Fld---------------------------------*/

void cblock_rega_fld_write(hwint32 write_data)

{

    hwint32 reg_read;

    write_data = write_data << CBLOCK_REGA_FLD_OFFSET;

    CBLOCK_REGA_READ(reg_read);

    write_data = (write_data & CBLOCK_REGA_FLD_MASK) | (reg_read & ~CBLOCK_REGA_FLD_MASK);

    CBLOCK_REGA_WRITE(write_data);

}

/*---------------------------- End of Write Function for field : Fld---------------------------------*/






Additional #defines when repeat is used


To provide flxibility to the user to calculate offset value of instances of section/register with repeat greater than 1, IDS has enhanced C headers with separate defines for count value and size of first instance of that section/register.

Note: No separate property or switch is needed for this functonality. In case user's testcase has repeat, then these defines will be genrated by default.

Relocatibility of macros and register structs




Figure: Flow diagram


  1. As block1 and test_Block are used multiple times in the top hierarchy but there will be a single definition for each block and these blocks will be reused at multiple levels.
  2. Each block contains multiple registers, so Macro for each register and section in scope of that block will be added with an argument “baseAddress” to reuse the block at different levels by passing the “baseAddress” as argument of the parent chip.
  3. Macros for each chip in register map will be added in “allMacros.h”
  4. Naming conventions of leaf block elements will be:

<IP/BLOCK_NAME><REG_FILE_NAME><REG_NAME>” or

<IP/BLOCK_NAME>_<REG_NAME>

cheader_add_regmap

This property will be used to place the same register at different view maps.

Possible values: {cheader_add_regmap=<map_name>:<baseAddress>}

Multiple maps can be provided using comma (,)


Effect of Property: Each map_name from the property will be added(prepend) in the name of Macros in the “allMacros.h” file and corresponding baseAddress will be added to the address and offset of the chip and block.


Property Usage: “cheader_add_regmap” property will be applied at top element or can be provided at “-top_property” switch in command line.


command line :


          <IDSbatch existing command line> -top_property “cheader_add_regmap=mainProc:0x2000000,debugProc:0x5000000;cheader_opt=true"


Note: This property will be effective when used with cheader_opt=true.

cheader_add_regmap_offset


This property is used with the “cheader_add_regmap” property, using this we can change the offset of the regmap generated by “cheader_add_regmap”. The syntax to use this property is uvm_add_regmap_offset=”<name>:<base address>”. It is work with -top_property "cheader_opt=true"


SystemRDL :


property cheader_add_regmap {type=string; component=addrmap;};

property cheader_add_regmap_offset {type=string; component=addrmap;}; 

property chip {type=boolean; component=addrmap;};

reg regA {

  field {}F1;

};

addrmap block_1 {

    regA regA;

};

addrmap block_2 {

    regA regA;

};

addrmap block_3 {

    regA regA;

};

addrmap chip {

   chip=true;

   cheader_add_regmap = "map1:0x40000";

   block_1 b1[2] @0x10;

   b1->cheader_add_regmap_offset = "map1:0x20";

   block_2 b2 @0x30;

   block_3 b3 @0x40;

};

addrmap top {

   chip=true;

   cheader_add_regmap = "map1:0x40000,map2:0x200000";

   chip c1 @30000;

   c1->cheader_add_regmap_offset = "map2:0x200,map1:0x300";

};

Support of mixed data type in Cheader output:


c_type is a hierarchical property which is used to support mixed data types in c-header alt1. For example, If the user has multiple registers and they want to change the data type of some registers then they can apply the “c_type” property on those registers. The switch -c_type “uint%d_t” is used to declare integer data type inside struct at block level in header output. Here %d represents the format specifier for an integer value. Using this switch declares uint%d_t uniformly for registers inside the block.


Example:        IDS-NG        IDS-Word        IDS-Excel        SystemRDL

Vertical reuse(Cheader)

(Compatible with IDS version 6.18.0.0 and above)



When a block is vertically reused in a chip/system environment (top level flow) then the block related Struct gets regenerated again by IDS at the chip/system level. To avoid this, vertical_reuse can be used. The value of this property can be "true" or "false". It is supported in alt-1 version of Cheader.
