IDS-Word


In the document editor, such as MS-Word, the IDS tool bar is used to create templates to be filled by the user.


     



System : System is a collection of multiple boards. IDS System template is as shown below:


                  


The System template of IDS consists of:


Name 

Description 

system

Name of the system

 

system description

Description of the system. Description can also contain the properties, having syntax {property1_name=value; property2_name=value} 

Ex:{coverage=on; clock_edge=posedge}. 

Note: properties should be separated by semicolon “;”.



Board : A board is a physical arrangement of the logic design including multiple chips, support circuitry and wiring. The IDS board fully justifies the above statement, as it can hold multiple chips inside it, which in turn contains the multiple components hierarchically. IDS Board template is as shown below :

 

 

The Board template of IDS consists of:


Name 

Description

board

Name of the board

board description

Description of the board. Description can also contain the properties, having syntax {property1_name=value; property2_name=value}  

Ex:{coverage=on; clock_edge=posedge}. 

Note: properties should be separated by semicolon “;”.



Chip : A chip is a major building block of an electronic system such as an ASIC or an FPGA. It is a semiconductor component which provides the memory, logic and virtually all other intelligent functions in an electronic system. It contains a large number of electronic circuits made up of Logic Gates. The IDS chip works on the same concept as it contains multiple blocks inside it. IDS Chip template is as shown below: 



 

The Chip template of IDS consists of:


Name

Description

chip

Name of the chip

chip offset

The displacement of the chip in the memory from the address of the container(for a chip the container is board). It can be specified in decimal or hex.

chip size

size of the chip

chip description

Description of the board. Description can also contain the properties, having syntax {property1_name=value; property2_name=value} 

Ex: {coverage=on; clock_edge=posedge}

Note: properties should be separated by semicolon “;”



Block : A design methodology has portions of large designs divided into manageable sections or sub-blocks that get created, represented symbolically, designed and then connected together when completed. This methodology allows different parts of the design to work in parallel. A block is such a set of memory modules and accessing circuitry. A single chip can have multiple specialized blocks of logic. The IDS block is exactly the same as it contains multiple registers, reg groups, memory etc inside it and thus helps to complete a part of the circuitry which involves multiple components. IDS Block template is as shown below:



The Block template of IDS consists of:


Name

Description

block

Name of the block, i.e. the string that identifies the element in the specification

block offset

The displacement of the block in the memory from the from the address of the container (for a block the container template is chip). It can be specified in decimal or hex

block size

Size of the block

block description

Description of the block. Description can also contain the properties, having syntax {property1_name=value; property2_name=value} 

Ex:{coverage=on; clock_edge=posedge}

Note: properties should be separated by semicolon “;”.

block constraint

Enum or constraint for the entire block is defined here. Enum is defined in curly braces “{}”  

Ex:{0:false,1:true}.



Register : A reg or register is a memory location that is part of a processor or an I/O device. Each bit or set of bits within the register controls some behavior of the circuit. The IDS Register is also the same but with some additional features and capabilities which makes it different and more user friendly. IDS Register template is as shown below:

 


The Register template of IDS consists of:


Name

Description 

reg_name

 A unique string that identifies the name of the register.

 Ex: Read_write_reg

offset

The address of the register in the address map is w.r.t the address of the container (for the register the container is block/section). If the user defines the offset then value in the offset is the displacement added to the memory location of the container. The value can be a decimal value or a hexadecimal value.

Ex: 10 or 0x10

external

The possible values of this property field can be "true" or "false". The default value of this field is "false". If this property is set to "true" then RTL implementation is not created by the IDS, instead appropriate ports are created so that the user can provide the implementation to that register. For more details refer to the next section(need to add to external register)

Ex: true or false 

size

This specifies the width of the register and the value specified must be a multiple of 8.

bits

Shows the width of the particular field of the register. 

Ex: 0:16

sw access(s/w)

This field indicates the policy of software access. IDS register can be accessed by external bus like AMBA, Avalon etc. or  proprietary bus.

Ex: rw or wo or rc or rs or wrc or wrs or wc or ws  or wsrc or wcrs or w1c or  w1s or  w1t or  w0c or  w0s or w1src or  w1crs or w0src or  w0crs or  r/w1c r/w1t or r/w0t or  r/w0t or r/wc or  r/w0c or r/ws  or r/w1s or  w0t 

hw access(h/w)

This field indicates the policy of hardware access. The IDS register can be accessed by the hardware within the block.

Ex: ro or rw or wo or na 

default

This is the default value of the field. A value for the default field is mandatory. The value can be a decimal or hexadecimal or octal or binary value along with the base specified. The decimal value can be given without specifying the base.

Ex: 12 or 'b0101 or 'hf or 'o12

description

A description is the functionality associated with the register. In addition to this, various properties can also be defined in this field. For a list of properties, see here.

Ex: 1. Description as This is a register named " Read_write_reg"

      2. Property as {reset_type = async}

Any text can be added here. The text can be formatted as well (e.g. Bold, italicized, colored etc.)

RegGroup/Section : A RegGroup is a set of registers. A Register or even a RegGroup can reside inside a RegGroup, but a RegGroup cannot be left empty. The RegGroup can be used to construct any arbitrary multidimensional register set. 

 



The Register group template of IDS consists of:


Name

Description  

section name

Name of the section, i.e. the string that identifies the element in the specification

section size

Size of the section

section offset

The displacement of the section from the address of the container (for a section the container template is block)

section count

Number of times all the contents in section are repeated

section external

True/1 if section is external

section description

Section description can be added here. This column can also contain the properties having syntax {property1_name=value;property2_name=value} Ex: {coverage=on; reset_type=async}.

end section

Section is terminated with “end section” tag. If section is used then this is mandatory. This tag should be included in the same column as  section_name


Note : Some output transformation may not support reg group inside reg group, so reg group must have at least one register. 



Enum: The Enumerated type is a data type consisting of set of named values called elements, members or enumerators of the type. IDS provides the capability of defining the enum for the code in two ways:


    • By using the enum template.
    • By defining the enums in the register's field description.



1. By using the enum template.

    



The mnemonics defined in the enum templates are being used to specify the values to other templates. Ex:  the mnemonic can be used to specify the default value of the register field or to the various properties like repeat, offset etc. 


The properties supported by IDS enum template are:


Property

Description

name

A unique string that identifies the name of the enum template. 

Ex: Enum_name

mnemonic name

A unique string that identifies the name of the mnemonic. 

Ex: ON 

value

The value of the mnemonic is specified in this field. 

Ex: 0 or 0xf

description

This field is used to give the detail description if required for the particular mnemonic.

Ex: Switch the functionality off



2.  Defining the enums in the register's field description.


IDS also provides the method to define the enum without using the enum template. This is done by specifying the mnemonics in the description field of the register.

Define: In IDS, a define is used to name arbitrary integer values. Any number of identifiers are used to name arbitrary integer values. IDS  template for the define is as shown below:


 


The properties supported by IDS define template are explained below:


Property

Description

define name

A unique string that identifies the name of the define name. The name of the define can have maximum 40 characters.

 Ex: NUM_PORTS

value

The value of the define is specified in this field. 

Ex: 'h16 , 2000

description

This field is used to give the detailed description if required for the particular define.

Ex: Number of ports

private

This field is used to modify the access scope of identifiers. By default all identifiers are public in nature. Assigning value '1' makes them private to that particular design module.

Possible values for this field are 0 or 1. 0 is for public and 1 is for private access.



IDS generates the appropriate code for these defines accordingly.


Turbo Generation


The Turbo button on the IDS-Word tool-bar generates all the outputs from IDS-Word without annotation back to the Word file. It works on the docx file directly without the Word API and hence is faster. This option is available in IDS-Batch CLI with the "-nocache" switch, which enables output generation without the intermediate cache files.



Stop Button to halt the generate process:


Users can stop the generate process by clicking on the stop button.
