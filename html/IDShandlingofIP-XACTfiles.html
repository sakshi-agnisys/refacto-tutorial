IDS handling of IP-XACT files



IP-XACT supported

Following versions are supported by IDesignSpec (IDS):

  • 1.4 (spirit consortium)
  • 1.5 | 1685-2009 (spirit consortium)
  • 1685-2014

IP-XACT files can be used to describe a component or a design. Components are connected in a design.

Referencing IP-XACT components


IDS can refer to IP-XACT components. The input format of the IDS file determines how the component is referenced.


Default referencing


IP-XACT component is referenced as a block in IDS chip. Another hierarchy inside IP-XACT component is treated as section present above registers. If offset (offset = 0x81000) is provided at the referred component, then absolute address of the addressBlock is the sum of addressoffset and offset provided at referred block{0x81000 + 0x0 = 0x81000}.

Import of IP-XACT


  • IDS can import and export IP-XACT files.
  • IP-XACT design can be imported. In IDS they become a ‚ÄúChip‚Äù by default.
  • IP-XACT component becomes a Block in IDS. Deep hierarchy becomes section-in-section.
  • If ‚Äú-ipxact_comp board‚Äù is applied,  then the IP-XACT component becomes a Board in IDS, memoryMap becomes chip, addressBlock becomes block.


      NOTE: ‚Äú-ipxact_comp‚Äú is supported only in IP-XACT-2014 and IP-XACT-1.5

                    - In IP-XACT-1.4 we only import registers which is inside memoryMap


Export of IP-XACT


  • ‚Äú-multi_out‚Äù can be used to create multiple files for each of the blocks.
  • While exporting with the ‚Äú-multi_out‚Äù option, a chip becomes a design and each of the blocks becomes a component. 
  • A Block in IDS is exported as a Component in IP-XACT.
  • Properties are handled using vendor extensions.


       NOTE:  supported  in IP-XACT-2014 ,IP-XACT-1.5 and IP-XACT-1.4

Signals/Ports in IP-XACT


By using ‚ÄùSignals‚Äù table of IDS, users can generate one or more bus interfaces and signals interfaces at IP level. It generates a "bus definition" and an "abstraction definition" file for the specified bus in the output directory with the name of the bus.


For example, if bus is PROPRIETARY then the "bus definition" and "abstraction definition" will be as PROPRIETARY_abstarction_defination.ipxact.xml and PROPRIETARY_bus_defination.ipxact.xml.

 

NOTE: 

  • Signals/Ports are supported  in  IP-XACT-2014 and IP-XACT-1.5 not in IP-XACT-1.4
  • Multiple bus domains are supported in IP-XACT-2014 and  IP-XACT-1.5  not in IP-XACT-1.4 

 

For using this feature:

1. Buses are defined as ‚ÄúuserDefinedBusName (Valid bus name)‚Äù in "Signals" template.

for example Bus1(AMBA-AXI), busIP (OCP) etc.

    



Valid Buses are: AMBA-AXI, OCP, AMBA-AHB, AVALON, AMBA-APB, AMBA3-AHB-lite, WISHBONE, PROPRIETARY.



       NOTE:

  • All signals and buses defined in ‚ÄúSignal‚Äù table are by default created at IP level.
  • If user does not want any signal defined in ‚ÄúSignal‚Äù table, to be created at IP level, using a property called ‚Äúexport‚Äù

 

  How to create multiple bus domain in IP-XACT OUTPUT using IDS:


      1. Buses defined in bus domain table, should be defined in "Signals" table with same user defined                              name.

Summary of command line options/properties for IP-XACT


  • {ipxact_compact=true}

To remove extra section hierarchy while referencing IP-XACT. Note that this property can be added using the command line switch -top_property

  • -ipxact_comp [block|board]

By default, IP-XACTis imported as top block, when applied with ‚Äú-ipxact_comp board‚Äù then the IP-XACT is imported as top board in IDS.

  • -if_ipxact : Generates separate files for each block.
  • -bus_comp<‚Äùcomma separated files‚Äù> : Specified component files for IP-XACT.

ipxact_decrease_offset property:


ipxact_decrease property decreases the offset value in ipxact file when it is referred by any input type file.


We ref two ip-xact in the rdl, first is test.ipxact.xml from which we ref the chip addressblock which has offset 0x100 and when ipxact_decrease_offset property applied then in html output the final decreased offset(original_offset - decrease_offset) come to the rf1 block, same thing is done with second block in which we ref the test1.ipxact.xml addressblock chip whose offset is 0x500 and after the ipxact_decrease_offset property applied it reflects in the html output with rf2 offset.     



 SystemRDL input:-

     

    property refpath { type = string; component = addrmap; };

   property ipxact_decrease_offset { type = number; component = addrmap; };

   

   addrmap ref_block{

      ipxact_decrease_offset = 0x50;   

      refpath = "test.ipxact.xml:chip";

   };


   addrmap ref_block1{

      ipxact_decrease_offset = 0x50;   

      refpath = "test1.ipxact.xml:chip";

   };


   addrmap my_ip{

      ref_block rf1;

      ref_block1 rf2;

   };
