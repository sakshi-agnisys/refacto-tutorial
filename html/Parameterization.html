Parameterization

Parameterization

(Compatible with IDS-Word version 3.14.0 and above)

(Compatible with IDS-Excel version 5.14.18 and above)


In IDS, user is able to specify various parameters in a define template typically at the top of the document and use them as macros at different places in spec. 


For example : default values, repeat, offset and hdl_path etc. These parameters specified in the document will be reflected in the outputs. 


Note that the earlier (prior to version 3.14) mechanism of creating defines will still work. So if ‘$’ is not used in the define table then ‘define is formed in the output and no parameter is created.


The following paragraphs show sample Parameterization in the spec and the generated outputs.

NOTE: 

  • The define template inIDS Word must have a root template, in other words, a block or chip, or board or system template must precede the define template. This tells the parser the scope of the parameters.
  • There shouldn't be any empty ROW between defines and params.
  • Define table must be on the top in Excel sheet before creating register specification.
  • All the four columns " define name, define value, define description and private" (as defined in the IDS_template sheet) are must with define table to generate defines and parameters.
  • Size of the parameters can be specified by declaring the name of parameter as $paramName[32] where $paramName is the name of the parameter and 32 is the desired width of the parameter



Output Of the Define Template


  • Parameters are created as follows:


                    parameter  num1              =  'h1000 ;   
                    parameter  str2                 =  “/root”;      
                    parameter  num3              =  45 ;      


  • Non-parameters (defines) are created as follows:



              `define  num5                         0x70     

              `define  str5                            /top     

                    `define  num6                            16     



Parameter specifications in IDS-Word


Note :  Use parameter in offset, repeat, hdl_path and default




Note that the define template is used after the chip template. The define template can be placed anywhere in the document, not necessarily immediately following the chip template.

Parameters in hdl_path








     


       



    

     



  • The hdl_path property can be set to any string value 


      Example :  {hdl_path = /top/dut/regA}


  • Sometimes it is necessary to use parameters in the hdl_path, in that case, define parameters and use the string concatenation operator ‘+’. 


      Example :  {hdl_path = $Top + /dut/regA}


  • Hdl path in IDesignSpec™ can handle register arrays and register files. 


  • By default if hdl_path property is set to string, automatically the appropriate number of iterators are added to the end of the string.


      Example :  if this hdl_path property is attached to a register with a repeat (register array) {hdl_path = /top/dut/regA} then it will translate into /top/dut/regA[%d] 


     For complex multi-dimensional registers user can specify the location of the iterators using special ‘#n’ syntax. When n starts from the outer most repeating structure.


    Example :  if this hdl_path property is attached to a register inside a register Groups (register array) {hdl_path = /top/dut/regGroup[#1]/regA[#2]} then it will translate into /top/dut/regGroup[%d]/regA[%d]


  • Params and iterators can be mixed to create virtually any complex hdl_path


Example :  {hdl_path = $top + /dut/regA[#1][#2] + _rd_data }

      • If it's false(0), means it can be overridden.

Override Property


The override property set to '0' prevents overriding of all parameters in referred document by top level document.

Changing the parameter at run time



The main idea of using parameters is to have the ability to change the parameters at run-time. How exactly that is done is specific to the different vendors. 



Mentor Graphics


User can change the parameter at run time by using the option –g (parameter default) and –G (parameter override) through the command line. 


% vsim -c –g<name>=<value> [working library].[file name]

                              Or

% vsim -c –G<name>=<value> [working library].[file name]



Synopsys


User can change the parameter at run time by using the option


-pvalues+<hierarchical_path_to_parameter>=<value> through the command line.


*Note -path must be slash("/") saperated


% vcs -pvalue+<hierarchical_path_to_parameter>=<value> [file name]



Cadence


User can change the parameter at run time by using the option


     -defparam <hierarchical_path_to_parameter>=<value> through the command line. 


         *Note -path must be dot(".") saparated.


% irun -defparam<hierarchical_path_to_parameter>=<value> [file name]

Use of ‘parameter’ to update the register field access, at run time


We have default existing sw and hw accesses for field definition. But they are statically unchanged but now we can overwrite our software access on run time using parameter.


 SystemRDL Example


`ifndef IDS_UDP

property coverage { type = string; component = addrmap | regfile | reg; };

`endif

addrmap Block #( boolean $My_val=1) {

  name  = "Block Address Map";

  coverage = "on";

  reg Reg { 

     regwidth = 16;

    field {

      hw = na;

      sw = r;

    } f6[15:8] = 8'h0;

    field {

      hw = na;

      sw = r;

    } F5[7:7] = 1'h0;

    field {

      hw = na;

      sw = $My_val?rw:r;

    } F4[6:6] = 1'h1;

    field {

      hw = na;

      sw = r;

    } F3[5:4] = 2'h0;

    field {

      hw = na;

      sw = rw;

    } F2[3:3] = 1'h0;

  };

  Reg Reg @0x00;

};
