Miscellaneous Properties

Introduction


  1. “swwe” and “swwel” are software access properties that provides a means of specifying commonly used software modifiers on register fields. These properties are applied at  field  of register.
  2.  “we” and “wel” are hardware access properties that can be applied to fields to determine when hardware can update a hardware writable field.
  3. "rsvdset" and "rsvdsetX" are field properties to set the value of reserve bits to '1' or 'X' according to the property applied.


    

  • swwe:

 

swwe implies software write enable. This property has precedence over the software access property in determining its current access state.

When swwe property is applied then the software write will depend upon a write enable signal being active high, this functionality is imparted by default in IDS, but in case, the value of swwe is false, the write enable will not be present and the new value will not be written on the field, therefore, it will retain the default value. For swwe equals to true, it will follow expression:

 swwe: Software write-enable active high (field = swwe ? new : current)

 

System RDL:


 Example

 

In the example shown below, the software access is rw (read-write) by default but since swwe=false property has been applied, the software access in generated RTL will be modified to sw=r (read-only) functionality.

 

addrmap block1 {

reg Reg1 {

   field { swwe = false;}F1[31:0] = 32'h0;

  };

  Reg1 Reg1;

};

  • swwel



swwel implies software write enable low. This property has precedence over the software access property in determining its current access state.

When swwel property is applied then the software write will depend upon a write enable signal being active low, in case, the value of swwel is false, the write enable will not be present and the new value will not be written on the field, therefore, it will retain its default value. In case the property value is true, then it will follow this expression:

 

swwel: Software write-enable active low (field = swwel ? current : new)


In the example shown below, the software access is rw (read-write) by default but since swwel=false property has been applied, the software access in generated RTL will be modified to sw=r (read-only) functionality.

The read value of all not explicitly defined fields is set to 1 if rsvdset is True, otherwise, it is set to 0.

rtl_port_suffix


This property is used to change the predefined suffix used in IDS generated RTL.


Syntax: - 


{rtl_port_suffix = <Predefined Suffix>:<Postfix string>}


Postfix string:- 


This postfix string will append in the suffix to the port name generated. It will replace the pre-defined suffix (LHS of ‘:’) with user-defined suffix (RHS of ‘:’). 


Eg:-


{rtl_port_suffix = enb:EN,in_enb:IN_EN,in: IN,r:R}



List of pre-defined suffix:-


     -  enb

  • Dynamic Array Support


Dynamically overriding the property at run time for a certain iteration in the register if count is applied on the register. It has following advantages:

  • User can change the default value of register at run time.
  • User can override the counter properties dynamically at run time.
  • User can override the resetsignal property at run time.

Support for removal of version/date information in verilog output

In the Verilog output, the header section typically includes information such as  "created by," "generated by," and "IDesignSpec ver." Profiler support in IDS-Batch


It has been observed that there is a need to improve performance and memory consumption in IDS-Batch. Moving some common output generation to JAVA from the earlier underlying technology, XSLT, has contributed to performance positively. Still, there are use cases where IDS-Batch takes a significant amount of time and consumes large memory. This feature is supported with the ‘-profile’ switch. This will also enable the ‘-verbose’ switch that will show each output generation start and end phase. -verbose switch provides start time and end time of chips/blocks and for different outputs. Another switch -log_verbose provides a way to dump this information in a separate file from the regular idsbatch log file. This implementation is a first step towards profiler support and needs to be extended to provide more information to serve the final objective. The profiler support is divided into two phases:


  • In the first phase, IDS-Batch captures the raw data in terms of the runtime for processing chip level and block level at different stages like annotated XRSL output, and all other outputs for design, verification and firmware. This captured data is dumped into a separate log file for performance analysis. In this phase user can see the amount of time and memory taken by a particular output and modules that can help to debug which is causing memory leakage and time consumption
  • In the second phase the solution will be extended to 
    • input SystemRDL(s) processing at the block level
    • capture memory usage at different levels in the RDL to output generation. 
    • We can architect annotation design divided by different phases like
      • Ref Resolve Processing
      • Instance expanding processing
      • Address calculation
      • DefineParamVariant resolution
      • Eval resolution 
      • UDP’s Resolution
      • Pre-Annotation Checks
      • Post-Annotation Checks
      • Quality checks
    • During each module processing for each output, there will be information of time and memory per output per module 
    • There is a command line switch named “-profile” that takes argument as “outputType:level”.example :- -profile “HTML:block;UVM:CHIP” default will be -profile “all:chip”




Support of -filelist switch for Verilog output

The -filelist switch effectively separates the `include files from the top wrapper file, relocating into a newly generated file. At present, IDS supports the -filelist switch for SystemVerilog(SV) output only.

Command Line: 

% idsbatch <input_file.rdl>  -out verilog  -bus <bus_name>  -dir ids -if  -filelist  “<name of filelist>”

When the -filelist switch is provided then the generation of the .f file will take place and all the `include directives from .v files will be removed.
