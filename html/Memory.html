Like all other IDS templates supports various properties, the memory template also provides the same capabilities. The various properties that can be defined for a memory template are explained below :



Property

Definition

name

A unique string that identifies the name of the system. 


  Example : Memory_name

offset

The address of the register in the address map is with respect to the address of the container. If the user defines the offset then value in the offset is the displacement added to the memory location of the container. The value can be decimal value or a hexadecimal value.


Example : 10 or 0x10

depth

The depth property specifies the depth of the memory.


In IDS-Excel, repeat value on a section determines the memory's depth.


Example : 10 or 0xf

width

The width property specifies the width of the memory.


In IDS-Excel, register width becomes width of the memory.


Example : 8 or 0xa 

default

The default property specifies the reset/default value of the memory.


Example : 1 or 0xa

In IDS, memories are conceptualized as external section (register group). This definition provides multiple approaches of defining a memory component in IDS flow.

1. External section with IDS property {type=mem}:


IDS property {type=mem} on an external section explicitly defines it as a memory component. Depth of this memory can be provided using property {depth=<positive int>}. Repeat on memory is provided using {repeat=<positive   int>}. Its effect on different IDS outputs:

  1. RTL:

In RTL separate decode logic is generated for aligned/unaligned memories. IDS currently support memories having width same as bus width. 

Limitation: -

        • Currently IDS do not support memory having width different than the bus width through direct access.
        • With {type=mem} only one register can be defined inside the external section. However multiple fields are allowed.
  1. UVM:

Memory object is extended from the uvm_mem class.

  1. HTML:

Memory object is created in the HTML output.

Limitation: -

        • With {type=mem} only one register with one complete field can be defined inside the external section. Multiple fields are not allowed.


2. External section:


Memory depth can be provided using property {repeat=<positive int>} and width is defined on the external section. Effect on different IDS outputs:

  1. RTL: 

Decode logic condition same as with {type=mem} for aligned/unaligned memories. Multiple Register with multiple field can be defined inside the external section.

Limitation: -

        • Currently IDS do not support memory having width different than the bus width through direct access.
  1. UVM:

Memory object is extended from the uvm_regfile class.

  1. HTML:

External section is created in the HTML output.


3.External section with IDS property {memory=all/rtl/uvm}:


Memory depth is the total number of registers in the external section and width is max width of these registers. Repeat on memory is provided using {repeat=<positive   int>}. Implemented to support structures inside a memory. One limitation, for large depth multiple registers must be defined. Effect on different IDS outputs:

a) RTL: 

Decode logic condition same as with {type=mem} for aligned/unaligned memories. Multiple Register with multiple field can be defined inside the external section.

Limitation: -

        • Currently IDS do not support memory having width different than the bus width through direct access.

b) UVM:

Memory object is extended from the uvm_mem class.

c) HTML:

External section is created in the HTML output.

Note : 

1. The memory cannot lie outside of the two to the power boundary i.e. The start address of a memory should have zeros in the lsb bits.The number of lsb bits should be as many as required to address all address locations in the memory/regGroup.For example : if size is 8, then 3 lsb bits should be 000 for the start address. 


2. By default the memories created by IDS are software read-only (ro) but the user can change them to software read-writable (rw) using "sw" property.

 

support of memory-width other than power of 2 and non-multiple of bus-width


The generation of various respected output for memory with memwidth nonpower of 2 and also non multiple of bus-width.

Case1:The memwidth 96 bit, which is non-power of 2, multiple of bus-width and 8 is being generated by considering only 96 bit.

Case2:The memwidth 97 bit, which is neither power of 2 nor multiple of bus-width and 8 would consider 104 bit wide memwidth, i.e. We would allocate one byte address for the value of memwidth for this.
