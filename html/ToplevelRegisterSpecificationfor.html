Creating Top Level Registers Specification for SoC


  1. In this document, we have listed down step by step process to create SOC’s top level registers specification by referring to block or subblock level registers specifications. 

  2. Board, Chip, Blocks, Subblocks, make the hierarchy. Using Microsoft Excel is a convenient mechanism to define the SoC hierarchy and that’s what we have used in various steps listed below.

  3. To understand, let’s start from the very basic block diagram. Consider Fig 1, Chip 1,2,3 constitute of multiple blocks 1,2,3 and these chips are further referenced in board 1 as Chip 1,2,3 etc.


                                                                                                                       Fig 1




  1. While Creating xlsx at board level, the next step is to write down the specification within a document at SoC Level by defining a board, chips, blocks and so on. The process to define the following can be accessed here Defining a Chip and Defining a Board.

  2. The addresses are calculated automatically if they are not mentioned explicitly. Any overlap detected at any level is flagged. 
    To check the overlapping of address, the command line option “global addressing = True” is needed in the IDS-Batch command. An example of the command line is mentioned in one of the steps below.


  1. The snapshot below defines 

    1. Board ->Board 1
    2. Chip -> Chip 1 alongwith chipsize and chip offset. If not given explicitly, the chip size and offset will be calculated implicitly.

      For instance in this case, the chip size 0x0100_0000 with chip offset 0xa100_0000 has multiple blocks – Block 1, Block 2 and Block 3 with mentioned size and offset.





  1. After defining the above-mentioned details, we can create subblocks. The process to define a sub-block can be accessed  here . 
    For instance, the Block 1 has multiple subblocks namely SB_1, SB_2, SB_3, SB_4, SB_5.

  1. After defining the subblocks, we need to create references to leaf level register specifications. The process to define a reference can be accessed here. 


For instance, Block 1 has multiple subblocks namely SB_1, SB_2, SB_3, SB_4, SB_5 with references as ref1, ref2, ref3, ref4, ref5  with ref paths mentioned as ref1_csr.xls:ref1_csr, ref2_csr.xls:ref2_csr,  ref3._csr.xls:ref3_csr,  ref4_csr.xls:ref4_csr,  ref5_csr.xls:ref5_csr.

  1. User can also define the reference path for the blocks which are at the leaf level. The process to define a reference can be accessed here. 
    For instance, Chip 1 has multiple blocks namely Block 2, Block 3 with references as ref6, ref7 with their respective ref paths mentioned as ref6_csr.xls:ref6_csr, ref7_csr.xls:ref7_csr.







  1. The block related UVM classes are regenerated by IDS at the chip/system level. But instead of regeneration, IDS has the capability to reuse the block related UVM class definitions generated at the block/IP level. A block’s UVM can be reused at top level.  To understand reuse, read here.

            Please Note : vertical_reuse=true will only work for out_third_party option, in case the users don’t wish to generate the UVM                     model again at board or chip level.

  1. IDS also has an option to generate hierarchical HTML files from the top-level specification. By default, a flat level single HTML is generated but if a command line option “-if” or “-if_html” is given on the command line, hierarchical HTML files are generated. Hierarchical HTMLs are generated only if “htmlalt2” is chosen as one of the outputs. This is very useful for large SoC datasheet generation. More Details

  2. Now after creating this xlsx at board level, we give this as an input to IDS-Batch using the IDS-Batch command. It checks for address overlap and generates the desired output as per the command line options given.

    Following is a sample IDS-Batch command-line which can be used at the top level to generate UVM and HTML outputs:



IDS-Batch Command:


idsbatch ABC level.xlsx -sheet_name Board  -no_formatting -output "uvm htmlalt2" -out_third_party d -hdlpath -coverage -addr_sort -if -top_property "global_addressing=true;uvm_lock_model=false;uvm.base_class=mvm_reg_block;vertical_reuse=true" -preserve -velocity "board:/cde/cad/tools/agnisys/IDSbatch/template/%board%_regdef.svh.vm board:/cde/cad/tools/agnisys/IDSbatch/template/%board%_struct.svh.vm chip:/cde/cad/tools/agnisys/IDSbatch/template/%chip%_regdef.svh.vm chip:/cde/cad/tools/agnisys/IDSbatch/template/%chip%_struct.svh.vm block:/cde/cad/tools/agnisys/IDSbatch/template/%block%_regdef.svh.vm block:/cde/cad/tools/agnisys/IDSbatch/template/%block%_struct.svh.vm"
