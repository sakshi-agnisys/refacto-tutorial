Enum

The Enumerated type is a data type consisting of set of named values called elements, members or enumerators of the type. The IDS provides the capability of defining the enum for the code in two ways:


1. By using the enum template.

2. By defining the enum in the register's field description.



The use of each of these method is explained below:

Enhancement for enum constructs flow instead of parameter constructs 


Enumerations in SystemRDL are not instantiated and can only be defined definitively (i.e., anonymous definitions are not allowed). Enumerated types can either be assigned to a field’s property or their enumerators can be referenced in expressions. For generating enum constructs for verilog output  in case of the usage of enums in the input test case and when enum encoding is used. A package of enum signals will be generated in a separate package file consisting of the enum constructs instead of the conventional parameter constructs. The above enum package file will be saved with .sv extension and will be included inside the register RTL file which will also be saved as a .sv extension file, to use enums in the register module.

The enum usage currently does not support enum constructs flow and gives parameter constructs instead in verilog files. By using a property “enum_package “ = true, the enum at present basically expects the enum constructs for the entire specification and generates the particular enum package file to be included inside the register file . The name of the RTL file must differ from the conventional sv output file just to make it distinguish from the other one.


SystemRDL Example


property enum_package {type = boolean; component = addrmap ; };

addrmap blocktop {

  name  = "blocktop Address Map";

  enum reg1_F1_e_enumName  { 

      en1=2'd1  ;

      en2=2'd2  ;

      en3=2'd3  ;

  }; 

  reg reg1 { 

     regwidth = 32;

    field {

     encode=reg1_F1_e_enumName ;

      hw = rw;

      sw = rw;

    } F1[31:0] = 32'd02;

  };

  reg1 reg1 @0x0;

};

Property ENUM_REMOVE 


Property enum_remove is used to delete unused enums from the RTL output. The possible value of this property = 'true' and by default it is always "false". This property is supported at the block level.

Support of “enum_remove” property in UVM


We have modified the property enum_remove to take comma separated values for outputs from which unused enums are removed, e.g. “enum_remove=uvm”  removes the unused enum typedef from the UVM output. IDS will also support this for Verilog in a similar manner(“enum_remove=verilog”) i.e. this will map with our existing property “enum_remove=true”).

If the user wants to remove the unused enum from verilog and uvm output both, then the user can use the property “enum_remove=uvm,Verilog”.


Note: A “enum_remove” property is supported at the top level element. Also, It can be applied by the  -top_property command-line switch.

Support of enum with values in the range from 1 to 5


We have now supported a property {cross=pending.F1: stat.F1:enable.F1} {desc= [uvm loc = top] cross.pending.F1:stat.F1:enable.F1 ignore_bins ignore_bin011 = binsof(pending_F1) intersect {0} && binsof(stat_F1) intersect {1} && binsof(enable_F1) intersect {1};ignore_bins ignore_bin101 = binsof(pending_F1) intersect {1} && binsof(stat_F1) intersect {0} && binsof(enable_F1) intersect {1}; [/uvm]} for the support of enums with values ranging from 1 to 5. The top property which will be used for this is "custom_code_hier=true". 


Input Register Specification:



SystemRDL Specification


addrmap Block2 {

  name  = "Block2 Address Map";

  ids_cross = "pending.F1:stat.F1:enable.F1";

desc = "[uvm loc=top]cross.pending.F1:stat.F1:enable.F1 ignore_bins ignore_bin011 = binsof(pending_F1) intersect {0} && binsof(stat_F1) intersect {1} && binsof(enable_F1) intersect {1};ignore_bins ignore_bin101 = binsof(pending_F1) intersect {1} && binsof(stat_F1) intersect {0} && binsof(enable_F1) intersect {1};[/uvm]";
