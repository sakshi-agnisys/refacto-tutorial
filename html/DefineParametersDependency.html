Define & Parameters Dependency


(Compatible in IDS version 5.14.18 and above)


In IDS, user is now able to specify parameters and defines in the define table typically at the top of the document and use these as macros to assign values to all properties (supported by IDesignSpec™)

Define/Parameter Creation


  • Parameter's name should starts with '$'
  • The value of the define/parameter can be static or based on an expression. Both String and numeric expressions are allowed.
  • Value can be an expression with the following operators:
      • Arithmetic operators : +, -, *, /, %, <<, >>, ()
      • Comparison operators/relational operators : >,<, >=, <=, ==
      • Conditional ternary operator: ?:
      • Concatenation Operator : + (overloaded in string context)


Rules for defining 'defines' and 'parameters'


  • 'Defines' should be defined on the top in define table before parameters.
  • Define could be dependent on already specified 'defines' only.
  • Parameter's value can depend on already specified defines/parameters.

Accessing Defines/Parameters


  • Syntax to use the defines in various properties- $define_name 


Example: 

      • clock_edge=$EDGE
      • reset_type=$isAsync?sync:async


  • Syntax to use the parameters in various properties- parameter_name


Example: 

      • repeat=$Count
      • offset=$address+$ROUTER_COUNT*$Chip_off



Note:

Following properties support only parameters for their value assignment-

      • repeat
      • field default
      • offset
      • hdl_path

Parametric RTL


IDS historically supports generation of parameterized RTL and these parameters can be specified in the specification itself. When the System RDL standard was launched and IDS started supporting System RDL as input there was a need to differentiate between IDS parameters and System RDL parameters. For this purpose, IDS parameters were prefixed with a “$” symbol. IDS users also adopted this convention although it wasn’t as per SystemRDL standard. IDS parser/compiler for System RDL is processing such parameters in a way that they are propagated to RTL outputs as RTL parameters. Any other System RDL parsers/compilers may not be supporting the IDS style parameters and would throw an error. 

Therefore, a property "define_as_param" can be used to generate parametric RTL from SystemRDL input. 

Parameters which are supported in parametric RTL are:

  1. Number of registers in an array of registers
  2. Field width
  3. Number of fields in a register
  4. presence or not of a register
  5. presence or not of a field
  6. presence or not of a regfile.
  7. presence or not of an addrmap (affecting UVM collaterals only).
  1. Number of regfiles in an array of regfiles.
  2. presence or not of a specific index regfile/addrmap within an array of regfiles/addrmaps


Used command line


idsbatch top_top.rdl -out verilog -bus apb -top_property "define_as_param=true" -dir output
