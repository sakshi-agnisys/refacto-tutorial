Application Logic Port Generation

Application Logic Port Generation in IDesignSpec™

Revision history

Version 1.0, July 8, 2019. Initial release.

Introduction

IDesignSpec™ is a powerful tool for entry of SoC register specification using GUI and/or text-based descriptions.  A register description may be entered in several formats including Microsoft Word (IDSWord add-on), IDS-Excel (IDSExcel add-on), IP-XACT, YAML, and SystemRDL (IDSBatch mode), or combinations of multiple formats.


This application note will describe the entry of several small, simple register descriptions where IDesignSpec™ will generate several ports that connect with the user application logic for each example.  The target audience is engineers with basic understanding of the IDesignSpec™ tool flow, and who wish to understand more about the inclusion of hardware-side ports generated by the tool, and where these ports connect to the application logic.  Some hardware-side ports are explicitly defined in a signal list, while others are inferred by the inclusion of a signal name in a property or where the port names are derived from register and/or register field names.


The ports that can be generated by IDesignSpec™ include:

  • Ports corresponding to register hardware-side access type.
  • Direct connections of explicitly defined signals, including secondary resets.
  • Clock-domain crossing clock inputs.
  • Ports needed to utilize RTL properties, such as enables and pulse outputs.
  • External registers connections (as specified by the user) where such external registers are the responsibility of the user to implement and connect to the IDesignSpec™ generated RTL.
  • Memory connections, where memory logic is external to the register RTL by definition.

Application Logic Port Generation by Source

Ports generated from Signals template


Specific ports may be specified in a Signals template.  The ports will be connected to registers and/or special logic via the use of properties such as “next”, “rtl_wrapper”, or “resetsignal”.


https://www.agnisys.com/release/docs/IDS/Signals.html


Logic Ports Generated for HW access type 


The choice of hardware access types in register descriptions will determine which ports are generated by IDesignSpec™ for each register/field.  There are 3 hardware access types: read/write (RW), read-only (RO), and write-only (WO) which describe how the application logic can access individual register fields.  These are the most common type of ports connected to application logic.


https://www.agnisys.com/release/docs/IDS/RegisterAccess.html


Clock-Domain Crossing clock ports


When the {cdc.clock=<clock_signal>} property is implemented in a field/reg in IDS-Word/IDSExcel or with (cdc_clock = "<clock_signal>") UDP is set in SystemRDL, the clock signal name is inferred from the definition and added to the port list in the HDL output.  This clock will be used to synchronize the output with the specified clock.  The user must connect this clock when instantiating the generated RTL block.


https://www.agnisys.com/release/docs/IDS/ClockDomainCrossing.html


RTL properties that generate ports


There are several RTL properties that create input ports to enable/disable HW updating of registers/fields, create output ports for register enables, and that create ports to output a pulse that occurs on specific read or writes to the registers/fields.


https://www.agnisys.com/release/docs/IDS/RTLProperties.html

External registers and memories


There is an option in register/block templates to utilize register or memories as external to the IDesignSpec™ generated RTL blocks.  IDesignSpec™ will create external ports to connect to these external register/memory implementations.

IDesignSpec™ considers all memories connected to the register map to be external and generates its corresponding hardware ports..  There is a memory template in IDS-Word as well as a memory header in IDS-Excel.


All external memories and registers in a block will share the same write_data input provided by the block.  However, each external component will receive its own write enable signal.  Memories will also be connected to an address bus, and it is advised to set an offset where the lowest address will be all 0’s (for avoiding the use of comparators during the address decode stage).


https://www.agnisys.com/release/docs/IDS/Memory.html

https://www.agnisys.com/release/docs/IDS/ExternalRegisters.html


The timing information of external registers is explained in the following:


https://www.agnisys.com/release/docs/IDS/TimingInformation.html

Generating HDL Outputs


In IDS-Word or IDS-Excel, the outputs are generated by selecting the desired file types in Configuration Settings (click on Configure) in the IDesignSpec™ ribbon under Registers > Outputs, clicking the “Check” button in the IDesignSpec™ ribbon, and finally clicking on the “Generate” button to produce outputs.  In this example the outputs are Verilog-1995 and VHDL-alt1, and the output directory uses the default name (IDS):



IDSBatch can accept several input formats, including previously created IDS-Word or IDS-Excel graphical representations, as well as text-based inputs such as SystemRDL (including version 2.0).  The IDS-Batch command line must specify a top-level argument, which in this case could be a chip or a block (e.g. “OurBlock”).  The following command line arguments will generate both Verilog-1995 using the AMBA-AHB bus: 


>IDSbatch OurBlockDesign.rdl  -top OurBlock -out “verilog” -bus amba -rdl_version 2

>IDSbatch OurBlockDesign.docx -top OurBlock -out “verilog” -bus amba

>IDSbatch OurBlockDesign.xlsx -top OurBlock -out “verilog” -bus amba

Signal template connected to register bits

The ports (bit_zero_out and bit_one_out) are connected to flip-flops created by the “next” property in IDS:

    always @(posedge clk)

        begin

        if (!reset_l)

            begin

                bit_one_out <= 1'b0;

            end

        else

            begin

                bit_one_out <= Direct_bit_bit_one_q;

            end

        end // end always

        always @(posedge clk)

            begin

            if (!reset_l)

                begin

                    bit_zero_out <= 1'b0;

                end

            else

                begin

                    bit_zero_out <= Direct_bit_bit_zero_q;

                end

            end // end always


For this example, the input signals reset0 and reset1 are used as a secondary, synchronous soft reset to the register OurReg_w_soft_reset.  The default is active-low if only the reset signal name is specified, but can be changed through specifying a property with additional arguments with four arguments separated by colons: signal name, default reset value (independent of the asynchronous reset value), type (async or sync) and level (low or high).  Additional soft resets can be specified with different levels and/or default reset values.

{resetsignal=<reset_signal0>:<d_val0>:<type0>:<lvl0>, <reset_signal1>:<d_val1>:<type1>:<lvl1>, ………….}

Note that the resetsignal property (with the same reset signal) may be applied to multiple registers and/or register fields.  Also, multiple external reset inputs may be utilized for a register and/or register field.

Summary


This application note has described various ports that IDesignSpec™ creates in RTL and that connect directly to application logic.  These signals are as follows, where explicit signals are named in a signals table, while inferred signal are created from register specifications or named as an argument for a property.


Signal type

Signal definition

Description

HW access ports

inferred

Hardware-side access ports are inferred from the names of registers and fields.

Direct connections

explicit

Explicitly named signals (in a signal table) may include direct connections (via a flop) to a register/field, or to secondary reset signals.

CDC clock input

inferred

CDC clocks are inferred as an argument of the {cdc_clock=<clock_name>} argument.

RTL properties

inferred

Several RTL properties infer signal names as a combination of register/field names.

External registers

inferred

External registers must be provided by the user.  IDS will create a standard set of ports to connect to these registers.

Memories

inferred

Memories are considered external by default and will be connected to a standard set of ports for memories.
