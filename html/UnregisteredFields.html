Unregistered Fields

Often designers require hardware status bits to be accessed by Software without being registered inside the IDS generated block. This functionality can be achieved by adding a property {registered=false} to the field . The default value of this property is "true". Following diagram shows how this property is interpreted and appropriate RTL structure is generated by IDS.


This property can also be added to the register. In this case, no internal register is implemented.





Hardware generated with unregistered field using IDS


Depending on the access of the software and hardware, any bits of a register can be accessed from software side without being registered inside the IDS.


Ex: If a register Reg1 has field F2 with accesses as sw=rw and hw=rw

RTL will be:  

assign Reg1_F2_q  =  Reg1_F2_in_enb ? Reg1_F2_in : (Reg1_wr_valid ? wr_data[15:8] : Reg1_F2_in) 


The hardware has the highest priority over the software. But it can be customized by using the property “rtl.precedence”


Sometimes designers have registers and fields with property {registered =false} and requires the fields to be cleared, set or toggled while reading, writing or both. In such cases, additional ports are generated rd_valid_out and wr_valid_out. These can be used for any trigger in the application logic.

registered = false:nomux(Supported from v6.8.10.0)


In 6.0 version of IDS when using property {registered = false}, an input enable signal gets generated. This enable signal creates a combinational feedback loop. 


When using property {registered = false:nomux}, enable signal is removed to get rid of feedback loop. This avoIDS timing issues and simpler logic in IDesignSpec™ version 6.8.10.0. With the use of ‘registered=false:nomux’ property, the mux which was earlier generated inside the IDS block, now will act as an external to the block.

Note:

_enb is an output from the memory map and pulses high when SW writes to the register. in_enb signals are inputs and *those* are for writing from hardware.
The example above generates the following ports in the IDS memory map under 6.0:
output   block1_IDSreg3_enb;

input  [31 : 0] block1_IDSreg3_fld_in

output [31 : 0] block1_IDSreg3_fld_r;

 input  block1_IDSreg3_fld _in_enb,

Under 6.8.10, the following signal is removed:


   input  block1_IDSreg3_fld_in_enb,



registered = false:retain


In IDesignSpec, the value of the unregistered fields can be retained using ‘retain’ argument in the existing “registered” property as {registered=false:retain}. Using this, an input enable signal (*_enb) is generated with the input hardware signal (*_in). This property generates a combination feedback loop with retain logic.


Examples:


IDS-Word




IDS-Excel



SystemRDL

`ifndef IDS_UDP

    property registered {type =string; component = reg | field ; };

`endif

 

addrmap Block1 {

       name  = "Block1 Address Map";


       reg Reg1 { 

               regwidth = 32;

               field {

                       registered = "false:retain" ;

                       hw = rw;

                       sw = rw;

                       onread=r;

                       onwrite=w;


               } Fld1[6:0] = 7'h0;

    

               field {

                       registered = "false:retain" ;

                       hw = rw;

                       sw = rw;

                       onread=r;

                       onwrite=w;

       } Fld2[31:7] = 25'h0;


  };

 

  Reg1 Reg1 @0x0;


};
