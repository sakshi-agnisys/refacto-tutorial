Quality Checks in IDS


QUALITY CHECKS FOR REGISTER-MAP


IDesignSpecTM accepts a register map and follows different defined checks for further processing over the captured register-map to control the output behavior. These set of defined checks now also include a battery of non-mandatory quality checks for an improved quality of a captured register map and to have a future proof design to avoid any issues at the later stages of the design development for smooth flow of the entire design process. They ensure that designs are easier to debug, optimized for improved performance, follows simplified hardware and firmware development, easier to work around defects at the final stages and all by anticipating the impact of the captured register specification on the entire design development cycle. 



The current continuously increasing lot of such quality checks include:

  • Checks for reserved spaces
  • Checks for appropriate access for the software interface
  • Checks for proper naming conventions
  • Checks for uniqueness
  • Checks in interrupts
  • Checks for parallel addresses
  • Checks for alignment



These quality checks which are associated with unique IDS can be enabled using the command line switch ‘-check_quality’ and failure in any of the quality check results in a detailed and descriptive warning along with a recommendation in the area of compromised quality of the register-map and a user may choose to either follow the recommendation or may ignore it (if its necessary obligation in the actual design) by using the switch ‘-no_warn’ and giving appropriate IDS as its argument . 


Example: 

Let quality checks enabled through the switch ‘check_quality’ in a captured design result in quality warnings with IDS – 10001 , 10003, 10010. In order to ignore the warning associated with IDS – 10003 and 10010 , the switch ‘-no_warn’ can be used to ignore such specific warnings as below:

%idsbatch <existing command line > -check_quality -no_warn “10003,10010”


Note: In IDS GUI formats/IDS-NG, these switches can be added in the ‘IDS-Batch CLI options bar’ at the bottom of the configuration window.


EXAMPLE:


System RDL:


addrmap Top_chip {

       name = "Top chip Address Map";


       addrmap Block1{

               name="Block1 Address Map";

               reg STATUS_REG { 

                       desc = "Status Register Counter";

                       regwidth = 32;

                       field {

                               hw = rw; sw = rw; onread=r; onwrite=w;

                       } CTR[31:0] = 32'h0;

               };


               reg ADDRESS_REG {

                       desc = "Address Register ";

                       regwidth = 32;

                       field {

                               hw = rw; sw = w;

                       } READ_ADDR[15:0] = 16'h0;


                       field {

                               hw = rw; sw = r; onread=r;

                       } WRITE_ADDR[31:16] = 16'h0;

               };


               STATUS_REG STATUS_REG @0x000000000;

               ADDRESS_REG ADDRESS_REG @0x000000004;

       };


       Block1 Block1 @0x000000000;

};


When IDS quality checks are run on the above captured legal register map (Figure 1), the following warnings and recommendations among the lot are currently displayed on the console over appropriate components of the specification –

  1. Warning on reserved spaces, as compact designs are prone to be backward incompatible when additional components are added at later stages of the design process or on re-use of the design in future projects.
  2. Warning on access types for the software interface as it is best recommended to use identical software accesses for all the bits of a register component to avoid confusion in writing and maintaining the code and reduce the risk of introducing errors in the firmware. Also, it is not recommended to use write-only bits as firmware cannot read it and has to maintain a copy to check for functional correctness.
  3. Warning on discontinuous bit allocation as adding continuous bits in a big-endian manner ensure firmware can access the bits with shorter hex values reducing complexity.  


The tool ensures that the warnings are descriptive and to the point and also leaves a pointer to the actual area of compromised quality in the entire specification.
