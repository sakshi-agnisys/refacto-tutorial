Dynamic Assignment Rules

Naming in case of multiple instances, parameterization, dynamic overriding of properties

 


As we all know, dynamic overriding and assignment of properties/UDPs is the most powerful feature of the System RDL standard. It can change the behavior of instances created for RDL components “addrmap/regfile/reg/field”.

Multiple instances of any of these collaterals are identical by default. Here, single collateral is generated for identical instances using the original name of the component. Using the dynamic assignment or overriding on any of those instances can make those instances unique. Here, separate collaterals are generated for each unique instance. But the question comes - “how do you name these individual instances?”. This appnote discusses the naming convention for multiple instances of an RDL component which has parameters and/or dynamic instances.


In case of parameterization, the names of the collaterals are generated based on the values of the parameters, this is enabled by the “inst_name_cppstyle=true” UDP because it's an approach similar to that used by C++ compilers. This was implemented in release 6.62.6.0 of IDS. For further details see this documentation page.


In case of dynamic assignments, it is possible to have an effect on the generated collateral and so the names of the generated collateral could potentially be different. Therefore the name of the instance is used to create the collateral. However, there are a few exceptions to the rule.


The exceptions introduced regarding the behavior of dynamic assignment/overriding for few properties/UDPs which are:

  1. name
  2. desc
  3. inst_name


Regarding the properties “name” and “desc”,

When the dynamic assignment/overriding will be done for both or one of “name” and “desc” properties then there will be no effect over output generation for C Header and UVM. 


Please refer the below example for the reference:


Case-I

RDL

addrmap my_memory {

  mem {

    mementries = DEPTH;

    memwidth   = WIDTH;

    sw = rw;

  } external memory;

};


addrmap mem_blk {

  chip = true;

  my_memory mem_inst_1;

    mem_inst_1->desc="This is name";

    mem_inst_1->name="This is desc";


  my_memory mem_inst_2;

    mem_inst_2->desc="This is name";

    mem_inst_2->name="This is desc";

};



UVM

Now

Two Collaterals are created for “my_memory”

First with Original Name

my_memory_pkg.regmem.sv

my_memory.regmem.sv

Second with Instance Name

mem_inst_2_pkg.regmem.sv

mem_inst_2.regmem.sv

mem_blk.regmem.sv

mem_blk_pkg.regmem.sv


After Enhancement

Only one Collateral will be created for “my_memory” with the original name

my_memory.regmem.sv

my_memory_pkg.regmem.sv

mem_blk.regmem.sv

mem_blk_pkg.regmem.sv


Header

Now

mem_blk.h

Two Collaterals are created for “my_memory”

First with Original Name

my_memory.h

Second with Instance Name

mem_inst_2.h

mem_blk_address.h


After Enhancement

mem_blk.h

Only one Collateral will be created for “my_memory” with the original name

my_memory.h

mem_blk_address.h


Notes:

  1. The above behavior is introduced because there is no effect of these properties on the logical information of the generated collaterals.
  2. The above convention is followed for rest of the components also “reg/regfile”.
  3. The above convention is irrespective of parameter overriding. In case parameter overriding is done with the above scenario then UVM and C Header output will be generated as Case-II mentioned below.

As you can see in the above scenario, dynamically overriding of “desc” and “name” properties do not affect the collateral generation. Unlikely, “inst_name” UDP will be handled smartly by the tool for the collateral generation in UVM and C Header output. 


Please refer the below examples for the reference:


Example Containing “desc”, “name”, and “reset” properties below:

RDL


reg reg1 {

field {}f1;

};


addrmap blk1 {

reg1 reg1_1;

reg1_1->name="This is name";

reg1_1->desc="This is desc";

reg1_1.f1->reset=1;

reg1 reg1_2;

reg1_2->name="This is name";

reg1_2->desc="This is desc";

reg1_2.f1->reset=0;

};


addrmap chip {

blk1 blk1;

};



UVM Collaterals

blk1_reg1

blk1_reg1_2


C Header Collaterals

blk1_reg1

blk1_reg1_2


Case-II

RDL

addrmap my_memory #(longint unsigned DEPTH = 256,

                    longint unsigned WIDTH = 32,

                    boolean ABC = false) {

  mem {

    mementries = DEPTH;

    memwidth   = WIDTH;

    sw = (ABC) ? r : rw;

  } external memory;

};


addrmap mem_blk #(string INST_NAME = "ABC",

                  boolean INTERNAL_ACCESS = true) {

  chip = true;

  my_memory #(.DEPTH(512*<%=$mem_kilo%>/4), .WIDTH(4*<%=$mem_byte%>))    

    mem_inst_1;

    mem_inst_1->desc="This is name";   //Optional

    mem_inst_1->name="This is desc";   //Optional

  my_memory #(.DEPTH(512*<%=$mem_kilo%>/4), .WIDTH(4*<%=$mem_byte%>))    

    mem_inst_2;

    mem_inst_2->desc="This is name";   //Optional

    mem_inst_2->name="This is desc";   //Optional

};


In this example, “my_memory” addrmap is instantiated twice including the parameter overriding. In this case, the parameter's values are the same for both instances. Due to which below collaterals are generated in UVM and C Header:


UVM

my_memory_131072_32_false.regmem.sv

my_memory_131072_32_false_pkg.regmem.sv

mem_blk_ABC_true.regmem.sv

mem_blk_ABC_true_pkg.regmem.sv


Header

mem_blk_ABC_true.h

my_memory_131072_32_false.h

mem_blk_ABC_true_address.h




Case-III

RDL

addrmap my_memory #(longint unsigned DEPTH = 256,

                    longint unsigned WIDTH = 32,

                    boolean ABC = false) {

  mem {

    mementries = DEPTH;

    memwidth   = WIDTH;

    sw = (ABC) ? r : rw;

  } external memory;

};


addrmap mem_blk #(string INST_NAME = "ABC",

                  boolean INTERNAL_ACCESS = true) {

  chip = true;

  my_memory #(.DEPTH(512*<%=$mem_kilo%>/4), .WIDTH(4*<%=$mem_byte%>))

    mem_inst_1->inst_name=INST_NAME;

  my_memory #(.DEPTH(512*<%=$mem_kilo%>/4), .WIDTH(4*<%=$mem_byte%>))    

    mem_inst_2;

    mem_inst_2->inst_name=INST_NAME;

};


This example is taken from Case-II and “inst_name” UDP is dynamically applied with the same value on both instances of “my_memory”. Below collaterals are generated in UVM and C Header output:

mem_blk_ABC_true_address.h

Updated Test Scenarios. 

If multiple instances are created from the same parameterized component, then those instances which have dynamic assignment, the component will be created with parameterized name and those without the dynamic assignment will be created with instance name.


Input:


reg reg1{

    field {}f1;

};

regfile sec1 #(string name="false")  {

reg1 reg1;

};


Case-I: 

RDL


addrmap my_memory {

    sec1  s1_inst; 

s1_inst.reg1.f1->reset=1'b1;

     sec1  s2_inst;   

};


UVM/CHeader

Current Generation

Two Classes/Structures are created for “sec1”

For First Instance

For Second Instance

my_memory_sec1_false


After Enhancement

For First Instance

For Second Instance

my_memory _s2_inst_false
