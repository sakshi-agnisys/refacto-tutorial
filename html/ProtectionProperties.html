Protection Properties

(Compatible with IDS version 5.16.0 and above, in Verilog RTL)


IDS checks the compatibility of transactions with respect to the register protection property(reg_prot) value. If it matches, then the transaction goes through else IDS generates the target error. We can say that if your register is secured, we cannot send an unsecured transaction else there will be an occurrence of target error. Similarly, if your register is privilege, we can not send unprivileged transactions. 

 

Following are the possible values for reg_prot 

  1. priv

  2. unpriv

  3. secure

  4. nonsecure

 

Also, multiple possible combinations of these values are allowed in this property. With the value "nonsecure", all transactions are considered to be valid. This is default  behavior of RTL generated by IDS.

 

1. The values "priv(privileged access)/unpriv(normal access)" are used to access PPROT[0]. "priv'' signifies the active high while "unpriv" signifies the active low.

 

2. The values “secure(secure access)/nonsecure(nonsecure access)" are used to access PPROT[1]. "nonsecure" signifies the active high while "secure" signifies the active low.

 

This UDP can have one or two values (comma separated). The encodings are in this way "priv, secure" & "priv, unsecure" & "unpriv, secure" & "unpriv, nonsecure" . "priv/unpriv" means "reg_prot" can have either "priv" or "unpriv", it can not be like this "priv,unpriv". This convention is the same for "secure/nonsecure" values.

1. In case of "reg_prot = priv",

 

    // Write Protection : Privileged Access

    assign Reg1_wprot = pprot_i[0];

 

    // Read Protection : Privileged Access

    assign Reg1_rprot = pprot_i[0];

 

    // Read and Write operation depends upon protection signal (active high)

    assign Reg1_wr_valid = Reg1_decode && wr_stb && Reg1_wprot;

    assign Reg1_rd_valid = Reg1_decode && rd_stb && Reg1_rprot;

 

    // Read and Write errors comes when protection signal is '0' in case of privileged access

    assign Reg1_write_error = Reg1_decode && wr_stb && ~Reg1_wprot;

    assign Reg1_read_error  = Reg1_decode && rd_stb && ~Reg1_rprot;

 

2. In case of "reg_prot = unpriv",

 

    // Write Protection : Unprivileged Access

    assign Reg2_wprot = ~pprot_i[0];

   

    // Read Protection : Unprivileged Access

    assign Reg2_rprot = ~pprot_i[0];

 

    // Read and Write operation depends upon protection signal (active low)

    assign Reg2_wr_valid = Reg2_decode && wr_stb && Reg2_wprot;

    assign Reg2_rd_valid = Reg2_decode && rd_stb && Reg2_rprot;

 

    // Read and Write errors comes when protection signal is '1' in case of Unprivileged access

    assign Reg2_write_error = Reg2_decode && wr_stb && ~Reg2_wprot;

    assign Reg2_read_error  = Reg2_decode && rd_stb && ~Reg2_rprot;

 

3. In case of "reg_prot = secure",

 

    // Write Protection : Secure Access

    assign Reg3_wprot = ~pprot_i[1];

   

    // Read Protection : Secure Access

    assign Reg3_rprot = ~pprot_i[1];

 

    // Read and Write operation depends upon protection signal (active low)

    assign Reg3_wr_valid = Reg3_decode && wr_stb && Reg3_wprot;

    assign Reg3_rd_valid = Reg3_decode && rd_stb && Reg3_rprot;

 

    // Read and Write errors comes when protection signal is '1' in case of secure access

    assign Reg3_write_error = Reg3_decode && wr_stb && ~Reg3_wprot;

    assign Reg3_read_error  = Reg3_decode && rd_stb && ~Reg3_rprot;

 

4. The case "reg_prot = nonsecure" is default  behavior in RTL generated by IDS.

        if (Control_Reg_F1_in_enb) //F1 : HW Write

            begin

                Control_Reg_F1_q <= Control_Reg_F1_in;

            end

        else

            begin

            if (Control_Reg_wr_valid) //F1 : SW Write

                begin

                    Control_Reg_F1_q <= (wr_data [31 : 16]  & (reg_enb  [31 : 16]  & Control_Reg_prot_mask [31 : 16] )) | (Control_Reg_F1_q & (~reg_enb  [31 : 16]  | ~Control_Reg_prot_mask [31 : 16] ));

                end

            end

        end

    end //end always

endmodule

         
Note: In the above RTL code a wire for register protection mask (Control_Reg_prot_mask) will be generated to have a mask of `prot` ored with reg's mask. The register mask will have 0's as value for the field which contains the property i.e. to have a secure transaction, rest of the fields not having the property will have 1 as their value.
