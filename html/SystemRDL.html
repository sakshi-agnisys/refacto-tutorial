SystemRDL

SystemRDL  


SystemRDL is a language for the design and delivery of intellectual property (IP) products used in designs. SystemRDL semantics supports the entire life-cycle of registers from specification, model generation, and design verification, to maintenance and documentation. Registers are not just limited to traditional configuration registers, but can also refer to register arrays and memories. 


IDesignSpecTM


IDesignSpecTM enables IP/SoC design teams to create code files from the register specification, saving time and producing consistent, quality results. IDesignSpecTM (IDS) generates System RDL registers right from the user's spec. This document describes what is generated and how the user can customize the generated output.

Note: Register Specification in word format or in SystemRDL format will generate the same outputs. It means either you can use SystemRDL or Word format to describe Register Spec. as an input to IDesignSpec™.


                                             



In the following sections you will see how to use word format to describe the Register Specification. Register specification in MS Word format is equivalent to that in SystemRDL format. Either format can be used to generate the other.    


Chip


     




Chip is the top level element of IDS, under which there is a hierarchy of blocks, which is explained below. The example above shows mapping of the IDS chip property to the corresponding system RDL addrmap. The properties offset, external and size are not used, as these don't have a corresponding mapping in System RDL, and therefore can be left empty.


Block 


Block is the next level hierarchy under chip. Blocks can have offsets, and the user can define the size of the block. Under blocks one can define regGroups(an array of registers) and simple registers.


   


The block entered in IDS results in an addrmap with a block name, e.g., ―ethernet‖ as shown: 



addrmap ethernet { 

  name = "blockname Address Map"; 

  reg regname { 

    field { 

      hw = w; 

      sw = rw;  

    } field1 [31:0] = 32'b00000000000000000000000000000110; 

  }; 

}; 

ethernet ethernet @0x00000C;

 

Blocks can be offset by putting in an offset value, and blocks can also have size. The block is instantiated, having Address @0x00000C generated from offset = 3 and +=0x800 generated from size =512 bytes in a 32-bit system. 

The property external is not used, as it has no corresponding construct in system RDL. 



Register 


Registers and RegGroups are defined in the hierarchy under blocks. Register is a combination of different fields. Fields can be defined as shown below. Fields have properties like bits, name,

sw access, hw access, default and description. Registers have an offset property by which one can specify the offset with respect to block or RegGroup. The external property can be used during RTL transformation. 


IDSWord


 



IDSExcel  







reg status_reg { 

  regwidth = 32;

  field { 

    hw = r; 

    sw = rw; 

  } field1[31:25] = 7'b0000111; 

  field { 

    hw = r; 

    sw = rw; 

  } field2[24:7] = 18'b000000000000000010; 

  field { 

    hw = rw; 

    sw = r; 

    reset = 3'b010;

  } field3[6:4]; 

  field { 

    hw = r; 

    sw = w; 

  } field4[3:0] = 4'b0001; 

}; 


Access Types 

Register can support all the access types supported by System RDL.


property uvm_base_class { type = string; component = addrmap | regfile | reg; };


addrmap top {


 uvm_base_class="my_base_class"; 

 reg reg1 {

   field {

    sw=rw;

    hw=rw;

   } b [31:0]=5;

 };


  reg1 reg1;  

}; 


RegGroup 

The next level in the hierarchy is RegGroup, which represents repetition of the contents it contains. RegGroup has same properties as register, but accepts repeat and size. The mapping of RegGroup to System RDL is addrmap. Addrmap can be inside another addrmap,so Regroup can be inside another RegGroup,and can have multi-level hierarchy.



IDSWord



IDSExcel  




  addrmap reggroupname {
    name = "reggroupname Address Map";

    reg Reg1 {

      regwidth = 32;
      field {
        hw = r;
        sw = rw;
      } Field1[31:0] = 32'b00000000000000000000000000000101; 
    };
        
    reg Reg2 {

      regwidth = 32;
      field {
        hw = r;
        sw = rw;
      } Field2[31:0] = 32'b00000000000000000000000000000100; 
    };
    Reg1 Reg1 @0x0008;
    Reg2 Reg2 @0x000C;
  };
  reggroupname  reggroupname[2] @0x0000 += 0x80; 

 

The Repeat property tells that how many times a particular register is repeated. In above example a regGroup containing two registers is repeated 2 times, so the system RDL output will include the same. 0X80 defines the actual size of the RegGroup; alternatively, the user can explicitly specify RegGroup size and the same appears inside the code. 


Another case of RegGroup is when RegGroup has only one register and has no offset defined. 


IDSWord


 


IDSExcel  



     
  reg regname {
    field {

      fieldwidth = 32;
      hw = rw;
      sw = rw;
    } Field1 = 32'b00000000000000000000000000000001; 
  };
 
  regname regname[3] @0x0000 +=0x4;


In this special case no addrmap is generated, so a register array is generated in system RDL. But if offset is defined in a single register then addrmap is generated.



Counter


When a field is defined as a counter, the value stored by the field is the counter’s current value. There is an implication of an additional input which shall increment/decrement the counter when asserted.

By default counters are incremented/decremented by one (1), but developers can specify another increment/decrement value. Since the validity or feasibility of an increment/decrement value is determined

by the field’s width (valid increment values can be of equal or smaller width than the field itself),


IDSWord

   


IDSExcel



  
  reg regname {


    field {
      hw = na;
      sw = rw;
      counter = true;
      incrvalue = 15;
      incrsaturate = 5;
      incrthreshold = 4;

    } Statusreg[31:0] = 32'b00000000000000000000000000000010; 
  };

  regname regname @0x0000;


In the above example field is defined as increment counter by property {counter=incr;}, other counter properties and there effects can be seen from the Properties section of this Document.

note:When field is defined as counter,h/w access must be na in IDS.


Interrupt

Interrupts can be used by properties such as intr.status, intr.enable, intr.detect, mask.enable. The information about these properties can be referred from the property section of this document.  

Example:

IDSWord

 

IDSExcel  



Note: s/w access of register declared as status must be r/w1c.


addrmap blockname {
  name = "blockname Address Map";
  reg statusreg {
    field {

      sw=rw;
      woclr = true;

      hw=rw;
      intr;
    } F1[31:0] = 32'b00000000000000000000000000000010; 
  };
  reg enablereg {
    field {
      hw = rw;
      sw = rw;
    } F2[31:0] = 32'b00000000000000000000000000000101; 
  };
  statusreg statusreg @0x0000;
  enablereg enablereg @0x0004;
  statusreg.F1 -> enable = enablereg.F2;
};

Unkonwn Default Reset Value of Register Field


By default, the reset value of fields shall be unknown, e.g., x in Verilog i.e the default value is unknown(X) when no reset is specified in the rdl specification. 


Example : 


addrmap block1 {

    reg reg1 {

      field {

        sw = rw;

        hw = rw;

      }f1=1'b0;

      field {

         sw = rw;

         hw = rw;

       }f2 = 1'b1;

       field {

         sw = rw;

         hw = rw;

       }f3 ; 

  };


reg1 reg1;


};


for the above example the reset value of field f1,f2,f3 are 0,1 and x respectively.

SystemRDL v2.0 :


IDS supports systemRDL 2.0 in output. Features such as :

  • Constraints
  • Memory
  • Software access
  • Parameters
  • Enum
  • HDL Path


have been implemented as per the systemRDL 2.0 standard. 


IDS also supports SystemRDL 2.0 as input. The following features are supported in systemRDL 2.0 as input:


Features:

  1. Parameterized documentation can be created. The parameters can be resolved at the documentation level.

For example: 


reg myReg #(longint unsigned SIZE = 32, boolean REGISTERED = true) {

regwidth = SIZE;

registered = REGISTERED;

field {} data[SIZE];

};

addrmap myAmap {

myReg reg32;

myReg reg32_arr[8];

myReg #(.SIZE(16)) reg16;

myReg #(.SIZE(8), .REGISTERED(false)) reg8;

};



  1. Structures are supported at documentation level:

     For example:

struct my_struct {

string foo ;

string desc1;

} ;


addrmap top {

regfile some_regfile #( my_struct arg ) {

reg {

desc = arg.foo ;

field {

desc = arg.desc1;


} a ;

} regA ;

} ;


some_regfile #(.arg( my_struct'{ foo: "reg desc", desc1:"field desc" })) regFA ;

some_regfile #(.arg( my_struct'{ foo: "reg2 desc", desc1:"field2 desc" })) reggrp2 ;


} ;


  1. Expressions are supported now with logical operations such as || , &&, !  etc.

For example:


reg some_reg #(boolean BYTEENB = false) {

ext_byte_enb = !BYTEENB ; //expression

field {} b;

b -> ispresent = false ;

field { ispresent = false ; } d ;

} ;

some_reg #(.BYTEENB(true)) reserved_reg ;

some_reg partially_reserved_reg ;

some_reg not_reserved_reg ;

not_reserved_reg.b -> ext_byte_enb = true ;

not_reserved_reg.d -> ext_byte_enb = true ;



  1. Values can be post assigned during declaration and values can be given in an array form:

For example:


addrmap block_def {

hdl_path = "int_block" ;


reg {

hdl_path = { “ext_hdl_path”, "externl_reg" } ;

field {} f1 ;

} external_reg ;


reg {

hdl_path = "int_reg" ;

field {} f1 ;

} internal_reg ;

} ;

addrmap top {

hdl_path = "TOP" ;

block_def int_block0 ;

int_block0 -> hdl_path = "int0" ; 

block_def int_block1 ;

int_block1 -> hdl_path = "int1" ;

};



  1. Memory can be defined using the keyword ‘mem’ as shown below :

For example:


mem fixed_mem #(longint unsigned  word_size = 32,   //mem keyword

         unsigned longint  memory_size = word_size * 4096) {

mementries = memory_size / word_size ;     //complex expressions supported

memwidth = word_size ;

} ;


  1. Different data types are supported and such as  longint unsigned, Boolean, string etc.


  1. Constraints are now supported in systemRDL 2.0 format

For example:

addrmap Admap {

  name = "Admap Address Map";


  reg  {

     regwidth = 32;

    field {

      constraint { this <=10; } Reg21_F1_constraint;

                                                                                                                                   

      hw = rw;

      sw = rw;


    } F1[16:0] = 17'h0;


  }Reg21;



  reg  {

     regwidth = 8;

    field {

      constraint { this == 8; } Reg32_F1_constraint;

         

      hw = rw;

      sw = rw;

      onread=r;

      onwrite=w;


    } F1[6:0] = 7'h0;


  }Reg32;

  

  };


  1. Enums are supported:

For example:


addrmap Admap {

  name = "Admap Address Map";

  enum enum_name {

      red=0 { desc = "abc";} ;

      blue=2 { desc = "xyz";} ;

      yellow=3 { desc = "123";} ;

  };



  reg  {

     regwidth = 32;

    field {

      encode=enum_name;

                                                                                                                                   

      hw = rw;

      sw = rw;


    } F1[16:0] = 17'h0;


  }Reg21;

  

  };




Note: By default, new SystemRDL compiler will work which is compliant to all the SystemRDL 2.0 standard . 

  • This compiler is very powerful and error messaging has been improved in it. 
  • The compiler is very fast, and the speed of generation has been improved upon.
  • The command for generation of the output is:


 IDS-Batch <filename.rdl> -out "verilog uvm" -top <topmodule_name>


Backward Incompatibality:


1. Constraint and hdl_path are keywords in sysetmRDL 2.0, therefore UDP declaration should be removed.


2. The default reset value of the field is changed from 0 to unknown(x). For maintaining backward compatibility users may use -top_property “default_reset = 0” in the command line.



For running the old compiler, command for generation of the output is given below:


  IDS-Batch <filename.rdl> -out "verilog uvm" -top <topmodule_name> -rdl_version 1




PERL PRE-PROCESSOR


Method 1:  


Following steps are must to be followed before running IDS-Batch:


1.    Update your PATH variable to include the path to asp-perl directory which is “<IDS-install-path>/apache-asp”

2.    Issue the following command to set another environment variable:

export IDS_ASP_PERL_PATH=<IDS-install-path>/apache-asp

(assuming you are using bash shell, please change the syntax appropriately if you are using some other shell)


The above mentioned steps would set the required environment for asp-perl.


Example:


addrmap top{

    field myField{};

    reg  myReg{ <% for( $i = 0; $i < 6; $i += 2 ) { 

    %> myField data<%=$i%> [<%=$i+1%>:<%=$i%>]; <% } %>

    };

    myReg r1; 

};


Note: In this method, normal IDS-Batch command line will be used with all the possible switches.


OR


Method 2: To generate System RDL file from perl preprocessor, please follow the steps below:


  1. Run "Perl Pre Processor" on the input rdl file.


  2. Use output of "Perl Pre Processor" as input to IDS-Batch.


  3. You can install the "Perl Pre Processor" utility from the link below:


http://pkgs.org/centos-5/repoforge-i386/perl-Apache-ASP-2.59-1.2.el5.rf.noarch.rpm.html


You can use the following command to run the perl preprocessor on System RDL file:

"asp-perl input_filename.rdl > output_filename.rdl"


This output of Perl Preprocessor (output_filename.rdl) is given as input to IDS-Batch.
