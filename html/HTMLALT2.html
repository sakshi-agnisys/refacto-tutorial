HTML ALT2

IDS supports two formats to generate HTML alt2 output:

1. Hierarchical format

2. Flattened format


This document also explains various features provided in HTML alt2 output.


  • Hierarchical Format


In HTML alt2 output, to create hierarchical multiple output files, user can either use “-if" or "-if_html” switch along with the -htmlalt2 output switch. This is very useful for large SoC data-sheet generation. For no switch, by default IDS generates HTML output in flattened format.

Controlling generation of signals, variants, enums and define tables


  • “doc.rm_signal”, “doc.rm_variant” and "doc.rm" properties are now supported for removing signal and variant tables in HTML-alt2 output. Multiple tables can now be removed by specifying multiple values in this property separated by a comma, e.g., “doc.rm=signal,variant,enum,define” property.


    • "doc.rm_signal": Used to control the signal table in top level whether to display it or remove it from html output. It accepts boolean value. For value as true, the signal table gets disabled in html output. For value as false, it displays the signal table. Default value of this property is false. 


    • "doc.rm_variant": Used to control the variant table in top level whether to display it or remove it from html output. It accepts boolean value. For value as true, the variant table gets disabled in html output. For value as false, it displays the variant table. Default value of this property is false.


    • "doc.rm=signal,variant,enum,define": Used to control the variant, signal, enum & define table. If multiple tables are to be removed, using this property removes the mentioned table from html output.

Sorting fields

IDS provides multiple field descriptions for registers and each field contains bit size. Field Description can be added in unordered way w.r.t bitsize in a register. The output of such a test case has the same order of field description as given in the testcase.

To sort such field descriptions w.r.t. bit size in HTML alt2 output, IDS provides a property field_sort, which accepts boolean input values and if the value is set to true in the top level of the test case then field description will be sorted in descending order w.r.t. bit size.

Following example shows a rdl test case with 4 field descriptions in unordered and field_sort property defined in top level.

Img tag support with HTML & PDF formatting customization with user defined CSS file

Cascading Style Sheets (CSS) is a style sheet language used for describing the presentation of a document written in a markup language such as HTML. CSS is designed to enable the separation of presentation and content, including layout, colors, and fonts. 

This separation has following benefits:

    • improves content accessibility
    • provides more flexibility and control in the specification of presentation characteristics,
    • enables multiple web pages to share formatting
    • reduces complexity and repetition in the structural content
    • improves page load speed between the pages that share the file and its formatting as .css file is cache


Command line: IDS-Batch <input_file_name> -out “<documentation_output_types>” -dir <output_directory_name> -switch <css_file_name>


Ex: IDS-Batch inputfile.rdl -out “htmlalt2 pdfalt4” -dir output_dir_name -html_css “html_modified.css” -pdf_css “pdf_modified.css”


Note: IDS will accept absolute path as well as relative path css file path.

Search components by address  in HTML


Users can conveniently search for components based on their respective addresses. This feature significantly improves the user experience by facilitating streamlined access to specific components within the HTML output. Address-based search feature is a default behavior. 


Advantages of Address-Based Searching

Searching components by address in HTML offers several advantages over the traditional search by component name:


Efficient Navigation:

Utilizing the hierarchical path to navigate to specific locations within the document enables users to make direct jumps to the desired component.

Time-Saving Process:

Direct access to components via their addresses significantly reduces the time spent searching for and locating specific components, enhancing overall efficiency.

By this feature in HTML outputs, aim is to optimize component navigation and enhance the overall usability of platform. Users can now leverage address-based searching for a more efficient and productive workflow.

Support for command-line option "-log_hermetic"

The -log_hermetic switch is a powerful enhancement that offers precise control over detailed logging. This feature combines a straightforward switch to enable or disable logging with the added capability to set a disable time, ensuring efficient and controlled log file generation.

The disable time feature, when activated, excludes timestamp information from the log entries. Timestamps typically indicate when an event or action occurred. By disabling timestamps, users can generate log files without the overhead of recording precise timing details. This can be beneficial when absolute timing information is not crucial, enhancing log readability and potentially reducing the size of log files.
