Clock Domain Crossing

(compatible with IDS version 5.14.0.0 and above for Verilog)

(Compatible in IDS version 6.0.0.0 and above for System Verilog)


Various IP blocks within an SoC are often required to work in different clock domains in order to satisfy the power constraints. Various techniques are used to avoid meta-stability as signals cross from one clock domain to another. In IDesignSpecTM (IDS), we support CDC from both software side and hardware side. 



CDC from hardware side


On hardware side, there are two ways of addressing Clock Domain Crossing (CDC) issues. These are:

  • CDC without handshake
  • CDC with handshake

In CDC without handshake, "cdc.clock = <h/w_clock_name>" property can be used at register and field.

In CDC with handshake, "cdc.clock = <h/w_clock_name>:handshake" property can be used at register level.


IDS, basically, support following flows to resolve CDC issues on the HW interface:

1. Mux Synchronizer

Here synchronizers are used in Hw input/output domain with respect to the IDS generated target. Flop synchronizer is used for synchronization in from Client to HW interface and MUX synchronizers are used for synchronization from HW interface to the client.

2. Handshake Synchronizer

Data can be passed between clock domains using a set of handshake control signals, depending on the application and the design requirement. When it comes to handshaking, the more control signals that are used, the longer the latency to pass data from one clock domain to another. The biggest disadvantage in using handshaking is the latency required to pass and recognize all of the handshaking signals for each data word that is transferred.


Fig 2: - Handshake Synchronizer


Write Operation

The hw write request (<>fld_in_enb) and data (<>fld_in signal) are asserted in the hw clock domain. The write_req_ff2 is then passed through a 2-flip-flop chain in the sw clock domain and after that its acknowledgement write_ack_ff travels back though a 2-flip-flop chain in the hw clock domain.  The write_req_ff0 drives the <>fld_in_enb_ff signal, on which the registered <>fld_in value (i.e. <>fld_in_ff) is updated inside the <>fld_q. During this write req-ack cycle the <>fld_busy_out signal is asserted to indicate the ongoing write transaction.


Fig 3: - Write Operation

Read Operation

An internal read enable signal is generated in the sw clock domain in case of a value change event on the <>fld_q. This read request is then converted into a toggle signal and passed across CDC through a 2-flip-flop chain. In the hw clock domain it is converted back and is used to put updated registered <>fld_q_ff on to the <>fld_r signal.

Similarly, the acknowledgement pulse is recreated sw clock domain after passing through a 2-flip-flop chain. To prevent data loss, during the read req-ack cycle an internal <>fld_read_busy would be set which restricts the bus to update the <>fld_q. So, the <>fld_q can be updated only when <>fld_read_busy is not high. In case a write transaction comes in between, the register bus will be stalled to extend the transaction until <>fld_read_busy is de-asserted. 

Fig 4: - Read Operation

3. Pulse Synchronizer

IDS support a single cycle pulse on the hardware interface with property {singlepulse=true} on a single bit register field. This register field must have its s/w access 'wo' and its h/w access must be 'ro'. In case of handshake synchronization flow a pulse-synchronizer logic is placed on this pulse signal to enable proper synchronization across the CDC boundary.  


SystemRDL


property cdc_clock {type = string ; component = field ;};

addrmap Block1 {

  reg Reg1 {

    field {

    siglepulse = true;

    cdc_clock = "hw_clk:handshake"

      sw = w;

      hw = r;

    }F1[0:0] = 0x0;

  };

  Reg1 Reg1 @0x0;

};

Example of Specifying CDC in IDesignSpec™


To add CDC to a field, simply add the "cdc.clock" property on the field. Set the property value to the clock in the HW interface domain. See example below. This property instructs IDS to automatically insert the Flop and Mux synchronizers in the output. If this property is added to the register, then all fields get synchronized.

Limitations


  1. Only HW interface can accommodate multiple clock domains.
  2. CDC feature can be applied to only registered fields or registers.



Areas for future Enhancements


Synchronization techniques described above are not limited to  just  two. There are other synchronization techniques which can be enhanced in future :

  1. Toggle Synchronizer
  2. Asynchronous FIFO



Custom CDC


In IDS, 2-FF chain directly in the IDS block or inside handshake synchronizer module, to synchronize required control and data signals on the HW interface. On Synthesizing RTL it is possible that the place-and-route tool can place these flops far apart thus causing metastability on the signals being synchronized and thus eliminating their purpose.


To resolve this issue the 2-FF chain for register clock and hw clock is put into separate modules named “agni_sync_sw_block” and “agni_sync_hw_block” and the current 2-FF logic is replaced with these modules in required synchronizers. This flow also allows users to connect their custom implementation for the FF chain. They need to instantiate their logic inside the “agni_sync_blocks”.


This enhancement will be incorporated in the IDS CDC flow using the new top property “custom_sync=true”. 

To preserve user logic in the “agni_sync_sw_block” and “agni_sync_hw_block” modules, “custom_sync” property will accept a colon (:) separated second argument, “no_generate”, which indicates the IDS to not generate the “agni_sync_block.v” file. By default IDS will generate the “agni_sync_block.v” file in each IDS run.

cdc.reset


Top property “cdc.reset” can be applied on specification having register fields with HW side CDC properties, {cdc.clock=<hw clock name>} and {cdc.clock=<hw clock name>:handshake}, to define a separate reset signal for all the HW clock logic in register block, synchronizers and “agni_sync_hw_block”. 

In case this property is not specified then the register block, synchronizers and both agni_sync_blocks will reset on the register bus reset, i.e.,  “reset_l”.

“cdc.reset” property can take following three colon (:) separated arguments:

  1. HW reset name : (Required) : Specifies the name of the HW clock domain reset signal. 
  2. Reset type : (Optional) : Specifies the type “sync/async” for the reset signal. By default the reset type is sync.
  3. Reset level : (Optional) : Specifies the level “low/high” for the reset signal. By default the reset level is low.

HW Write pulse synchronizer


Write pulse synchronizer is used to synchronize hardware events on a single width data in line (<reg>_<fld>_in), into the register field that does not have the field hw write control signal (<reg>_<fld>_in_enb). Handshake synchronization approach is used to allow synchronization from a fast HW clock to slow register clock.


A simple example of it can be synchronizing a status event into a single bit field having IDS property “rtl.hw_enb=false” (this property disables the generation of hw write control signal <>_in_enb). 


This enhancement will be incorporated in the IDS CDC flow for hw writable single bit fields having “rtl.hw_enb=false” and “cdc.clock=<hw_clock>:handshake” property. In case this field is SW writable  then an extra property “rtl.precedence=sw” also needs to be added.  


With the earlier proposed top property “custom_sync=true”, the 2-FF chain in the synchronizer will be replaced by “agni_sync_block”.

            status_F1_q <= 1'd0;

        end

    else

        begin

        if (status_wr_valid) //F1 : SW Write One To Clear

            begin

                status_F1_q <= status_F1_q & ( ~ ( wr_data [1]  & reg_enb  [1] ));

            end

        else

            begin

                

                status_F1_q <= status_F1_in_ff;

            end

        end

    end //end always

    

    handshake_synchronizer_write_pulse #(.field_width(1)) status_F1_write(.clk(clk),.reset_l( reset_l),.hw_clk(my_clock),

    .register_fld_in(status_F1_in),

    .register_fld_in_ff(status_F1_in_ff),

    .register_fld_busy_out(status_F1_busy_out));

    assign status_rd_data  = status_rd_valid ? {30'h0, status_F1_q, 1'h0} : 32'd0;

   . . . .

   . . . .

   . . . .

   . . . .

endmodule



Note: It is assumed that no event will come while the synchronizer is still busy handing the last event, otherwise it will be missed.


CDC from software side


Sometimes it is required that the complete block operates on a clock which is different from the bus clock. This can be done by using property “reg_clk = <clock_name>” which is applicable on block, chip, board, and system.

This property will create an input clock on which read and write operation on register will take place and also instantiate custom synchronizer widget. Now the transaction coming from the bus will first get translated into custom and then will be fed to the custom synchronizer. Custom synchronizers will do the CDC using the handshake method and then provide read write transaction for the registers.

Support of SW-based CDC for AMBA AXI-based bus


Currently, the generation of CDC from the SW side gets the handshake module name instantiation which is same for all the buses used.  Through this enhancement we have provided different module names of instantiation in the generated code when the software side CDC is used.

In the case of the AXI bus, we have introduced the instantiated handshake module name to be handshake_axi_widget. For the remaining buses, it would be by default handshake_widget.


For AXI bus


handshake_axi_widget #(.addr_width(addr_width), .bus_width(bus_width), .prot(3)) cdc_sw(

            .clk2(sw_clk),

            .widget_prot_i(pprot_i),


For APB bus/All buses


handshake_widget #(.addr_width(addr_width), .bus_width(bus_width), .prot(3)) cdc_sw(

            .clk2(sw_clk),

            .widget_prot_i(pprot_i),


Limitations


  1. Only HW interface can accommodate multiple clock domains
  2. CDC feature can be applied to only registered fields or registers
  3. CDC with handshake is applicable at registers only
  4. CDC from software side is supported with ‘-fast’ switch
  5. CDC from sw side is supported with only axi bus for VHDL output. 


Note:  

  1. CDC is supported with Tilelink Bus interface from version 7.12.0.0 onwards.
  2. CDC is supported for VHDL output from version 7.16.0.0 onwards.
