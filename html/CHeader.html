C Header

C Header files are generated by IDS for the purpose of inclusion in C/C++ code. These header files enable engineers to get access to names, locations and various properties of the registers and bit fields. The C headers are created in a variety of ways in order to suite a wide variety of customer requirements. 


Alt1 : Alternate 1 : This file contains structures and unions used to create an exact representation of the Register model.

Alt2 : Alternate 2 : This file contains macros and #define preprocessors but devoid of any structures/unions for registers.

Misra-C : This file contains C headers conforming to the MISRA standard. 


  • Enum Name Format


The property cheader.enum.name_format is applied on the top block and provides the hierarchical name of enum according to the format specified in the property value in case of C header (alt1) outputs. 


{cheader.enum.name_format=%c%b%s%r%f%e,camelcase}


Detailed description of the property: -


cheader.enum.name_format : %c%b%s%r%f%e , camelcase


camelcase: To generate the enum name in camel format.


• %c : Here 'c' is referring to CHIP name. It will add the chip name in the enum name generated.

• %b : Here 'b' is referring to BLOCK name. It will add the block name in the enum name generated.

• %s : Here 's' is referring to SECTION/REGGROUP name. It will add the section/reggroup name in the enum name generated.

• %r : Here 'r' is referring to reg name. It will add the register name in the enum name generated.

• %f : Here 'f' is referring to field name. It will add the field name in the enum name generated.

• %e : Here 'e' is referring to enum name. It will add the enum name in the enum name generated.


  • "rsv_name_format" property


The property, ‘rsv_name_format’ is applied on the top (chip/block) to change the names of reserved field names in C-header alt1 output according to the format specified in the property value in case of C header (alt1) outputs. 


Example:

{rsv_name_format=%l<string>%h}


Following specifiers can be used alongwith the <string> in the value of the property:

  • %l -> lowoffset                                        
  • %o -> offset
  • %h -> highoffset

  • “description” property


This property named “description” can be used to shrink or suppress the descriptions of particular entities in case of C header (alt2) outputs.


It can be applied at the top element of the IP {e.g., chip for a chip level testcase or block in case of a block level testcase} as follows: 


{description=” chip: false, block: false, section: false, reg: false, field: false”}


The value of the property is by default ‘true’ for all entities (chip/block/section/register/field) unless specifies ‘false’ using the said property which removes the description of that particular entity in the C header alt2 output.

e.g.,


To suppress the description of section and register in a specification the property “description” can be given in the top element as: –

{description=” section: false, reg: false”}

  • "cheader_no_struct" property


These header files enable engineers to get access to names, locations and various properties of the registers and bit fields. IDS generated headers contains a union of structures and defines for components. This property is created to support Assembly language platform. In Assembly language, union of structure is not supported. By applying the property i.e. "cheader_no_struct", IDS will generate header output with only defines. This property can be applied at top level(block/chip) only. The value of this property can be true/false. By default, it will always false.

  • "cheader_guard_nameformat" property


By using this property, user can add a prefix(string) in the naming convention of guard define in header output. This property can be applied at top level(block/chip) only. The value of this property can be any string with or without %p. 

cheader_guard_nameformat = <Prefix String> <%p> <Postfix String>

where: %p - Name of Present Node (Top Element chip/block). <Prefix/Postfix> – Any String.

  • "cheader.initialization" property


Initialization of registers with default value helps in reading default value of register in reset condition. This will be implemented using property.


{cheader.initialization = true/false}


By default the value will be false. When applied on the top with value true will initialize all the registers in the hierarchy.

  • Maximum architecture size (in C header alt1)


IDS generated header alt1 files now considers maximum software architecture size while generating register structures/unions which allows accessing of large register definitions (register definitions greater than maximum software architecture size) in an array of size equal the maximum architecture size. 

This can be achieved through the command line switch  “-arch_size” whose value can either be ‘32’ or  ‘64’  to define maximum software architecture size as ‘32’ or ’64’ respectively.


Note- 

The usage of switch “-arch_size” will have precedence over the switch “-c_type” (if used)  in the generated register structure/union definitions.

  • [cheader loc = top]


IDS provides capability of adding custom blocks in the generated header alt1 code by using property, [cheader loc = <loc>] <custom code>[/cheader]. Location (<loc>) of this property could be head, tail or top. This property is supported from IDS v7.10.0.0 onwards.

Following example shows the usage of property with ‘top’ location.

c_type_std 


c_type_std switch is supported to change the data types according to the argument of the switch. With the “gcc” option, the data types will be changed to uint. For 32, 64 bit registers the data types will be uint32_t and uint64_t respectively. For 128 bit registers the data type will be “__uint128_t”. 


Also the standard library “<stdint.h>” will be included in the top file.

Support of Volatile Qualifier for Cheader output

In programming, the volatile qualifier is used to declare a variable whose value may change unexpectedly due to external factors that are beyond the control of the program.

The volatile qualifier ensures that every read and write operation on the variable is directly performed on the memory location associated with that variable, rather than relying on cached values in registers or applying certain optimizations that could lead to incorrect behavior. When a variable is declared as volatile, it informs the compiler that the variable's value can be modified by external entities or by other threads in the program. This means that the compiler should not make certain optimizations that could assume the variable's value remains unchanged. Currently IDS supports the volatile qualifier only for fields but with this implementation we are supporting it on register level also. 

Let's take a look at an example to understand this. Consider the following input:

      hw = r;
